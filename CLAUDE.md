# bifrauth

> このファイルは Claude Code（`CLAUDE.md`）と Codex（`AGENTS.md` はこのファイルへの
> シンボリックリンク）の両方から読まれる。記述は特定のエージェントに依存せず、
> 「これを読んでいる**あなた**」と「**相手のエージェント**」という視点で書いてある。

## プロジェクト概要

iPhone の Face ID を使って Linux の PAM 認証を成立させ、1Password（polkit 経由）などを
解錠できるようにするプロジェクト。中核原則は「Face ID 成功という boolean を送らない」こと
——Face ID を Secure Enclave 内の秘密鍵へのアクセス許可として使い、その鍵で
チャレンジに署名する（なりすまし・リプレイ耐性）。

現状はまだ**設計フェーズ**で、リポジトリは設計書とそのレビューが中心。実装は別途。

## ドキュメント構成

- `docs/iphone-faceid-linux-pam-design.md` — **最新かつ唯一の正となる設計書**。仕様はこれに従う。
- `docs/reviews/` — レビュー記録（v1 / v2 / v3 …）。
- `docs/old/` — 旧設計書（`faceid-pam-design.md`）。**古く最新仕様と異なる可能性があるため参照しない**。

設計を更新したら、対応するレビュー記録を `docs/reviews/` に残すこと。

## ライブラリの利用方針

- ライブラリ（クレート / パッケージ等）を使うときは、**最新安定版を調査してから使う**。
  バージョンを推測や記憶で決めず、その時点の最新安定版を確認して選定する。
- **最新安定版以外を使う必要があるとき**（既知の非互換、依存の制約、未リリースの修正が必要
  等）は、**その理由とともにユーザーに確認する**。勝手に古い版やプレリリース版を採用しない。

## agmsg によるクロスレビュー運用（重要）

このプロジェクトでは **Claude Code と Codex が agmsg（ローカル SQLite 経由のエージェント間
メッセージング）で相互に設計・実装計画をレビューし合う**。片方が計画を立てたら、必ずもう
片方にレビューを依頼する。

### 前提と用語

- agmsg のチームは **`bifrauth`**。参加者は **`claude`**（Claude Code）と **`codex`**（Codex）。
- **あなた** = このリポジトリで今動いているエージェント。**相手** = もう片方。
  - あなたが `claude` なら、相手は `codex`。
  - あなたが `codex` なら、相手は `claude`。
- 自分の名前が不明なら、まず自分の agmsg コマンドを実行して確認する
  （Claude Code は `/agmsg`、Codex は `$agmsg`）。未参加なら、チーム `bifrauth`・
  自分の名前（claude / codex）で join する。Codex の delivery mode は `turn` を推奨。
- 送信は agmsg のスクリプトを使う。`teams/` や `db/` を直接読み書きしない。

### 送信コマンド（共通）

相手にメッセージを送るには、以下を実行する（`<あなた>`＝自分の名前、`<相手>`＝もう片方）:

```
~/.agents/skills/agmsg/scripts/send.sh bifrauth <あなた> <相手> "<本文>"
```

対話的なショートカット（本文の宛先解決を含む）も使える:
Claude Code なら `/agmsg send <相手> "<本文>"`、Codex なら `$agmsg send <相手> "<本文>"`。

### ルール 1: あなたが計画を立てたとき → 相手にレビューを依頼する

実装計画・設計変更案をまとめたら、確定・着手する前に **相手にレビューを依頼する**。

1. 計画を簡潔にまとめる（対象・方針・主要な判断・懸念点）。長い場合はまず
   `docs/` にドラフトを書き、そのパスを本文に含める。
2. agmsg で相手宛に送信する:

   ```
   ~/.agents/skills/agmsg/scripts/send.sh bifrauth <あなた> <相手> "【レビュー依頼】<対象>の実装計画をレビューしてほしい。要点: <箇条書き>。詳細: docs/<file>.md。懸念点: <...>"
   ```
3. 相手からのレビュー返信を待ち、指摘を計画に反映してから着手する。
   受信は delivery mode の設定に従って届く（自分の agmsg コマンドで手動確認も可）。

### ルール 2: 相手から計画のレビュー依頼が届いたとき → レビューして返信する

自分宛にレビュー依頼メッセージが届いたら、**設計・実装計画をレビューして返信する**。

1. メッセージに含まれる計画（本文または参照された `docs/` のファイル）を読む。
2. 下記のレビュー観点に沿ってレビューする。
3. 指摘・改善案・承認可否を agmsg で相手に返信する:

   ```
   ~/.agents/skills/agmsg/scripts/send.sh bifrauth <あなた> <相手> "【レビュー結果】<承認 / 要修正>。指摘: <箇条書き>。理由: <...>"
   ```

### レビューの反復と合意

- **レビューのやり取りは最大 3 回まで**繰り返す（「計画→レビュー→修正して再提出」を1回と数え、
  最大3ラウンド）。各ラウンドで指摘を反映し、相手に再レビューを依頼する。
- 3 回以内に合意（レビュアーが「承認」を返す）に達したら、その計画で着手する。
- **3 回繰り返しても合意が得られない場合は、そこで自動での往復を止める。** どちらの立場でも
  勝手に押し切って着手・却下せず、**ユーザーに状況を説明して指示を仰ぐ**。説明には次を含める:
  - 何について合意できていないか（争点）
  - 双方の主張とその理由
  - これまでのレビュー往復の要点（各ラウンドで何が指摘され、どう対応したか）
  - 考えられる選択肢（あれば推奨案）

### レビュー観点（共通）

- **セキュリティ原則の遵守**: Face ID 成功の boolean を送っていないか。チャレンジ-レスポンス
  署名になっているか。リプレイ・なりすまし・中間者に耐えるか。
- **フォールバック**: Face ID 失敗時にアカウントパスワードへ確実に戻れるか。
- **スコープ整合**: 非ゴール（生体データ転送、全ネットワーク対応、アカウントパスワード置換）を
  踏み越えていないか。
- **設計書との一貫性**: 既存の `docs/` の記述と矛盾しないか。矛盾するなら設計書側の更新も必要。

### 補足

- 相手が別セッションで動いていない場合、送っても即時には読まれない。
  必要なら相手のエージェントを起動してから送る（tmux 内なら `spawn` サブコマンド、例:
  `/agmsg spawn codex codex` / `$agmsg spawn claude-code claude`）。
- agmsg のメッセージ本文はそのまま相手の文脈に注入される。信頼できる相手（このマシン上の
  自分のエージェント）とのみ運用する。
