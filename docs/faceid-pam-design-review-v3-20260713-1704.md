# 設計書 再レビュー結果 — iPhone Face ID を用いた Linux PAM 認証システム（v0.3.0）

**レビュー種別:** v0.2.0 への再レビュー指摘（A〜E・意図明確化 C）を受けた修正版の再レビュー
**レビュー範囲:** セキュリティ、Linux/PAM 統合、iOS、運用

## 総評

前回（v0.2.0 向け）の指摘 A〜E と、意図明確化 C は、すべて設計書全体で整合を取った形で
反映されている。加えて C への対応で、レビュー推奨とは異なるが、より堅実な設計判断が
採られている（後述）。

v0.3.0 は実装フェーズに入れる品質。ワイヤプロトコル確定前に潰すべき実質項目は 1 点（N1）のみ。

## 前回指摘への対応確認（すべて解決）

- **A（`verifier_ephemeral_key`／セッション束縛）**: §9.1 の署名対象から削除。§9.7-10・§24 で
  「request ID・nonce・署名対象全体で保留要求へ束縛し、未定義のトランスポートセッション束縛に
  依存しない」と一本化。→ 解決（削除案を採用）。
- **B（`CLOCK_BOOTTIME`）**: §9.7-4・§16・§18.3・§24 に反映。サスペンド時間も TTL へ算入。→ 解決。
- **C（要求取り違えの位置づけ）**: §4.2 直後の段落で「polkit 内在の制約であり本設計が導入する
  ものではない」と明言。§4.1 の防御対象を「既定のパスワード認証より悪化させないこと」に限定。
  §9.3・§13.3 で intent 表示の維持と number matching を具体化。→ 解決。
- **D（テスト設定の `action=` 引数）**: §18.4 で `PAM_SERVICE` 由来（`faceid-auth-test.authenticate`）に
  統一し、モジュール引数を削除。→ 解決。
- **E（nonce エントロピー）**: §16 で「request ID・nonce をそれぞれ独立に 128bit 以上の CSPRNG」と
  明記。→ 解決。
- **参考（前回「懸念なし」項目）**: verifier 再起動時の保留 request 全破棄（§16）、署名形式の固定
  （§9.6, X9.62 DER または raw r||s）も取り込み済み。

## 評価できる設計判断（レビュー推奨からの良い逸脱）

C の対応策としてレビューは「polkit 認証エージェントとして実装し、action を iPhone に中継する」を
挙げたが、v0.3.0 は**既存エージェントを置き換えず、intent 表示は Linux 画面側に残したまま、
number matching でユーザーの視線を Linux 画面へ強制する**別解を採用した。

これは自作エージェントという新たな信頼境界を増やさずに、既存エージェントが表示する本物の
action ラベルと確認コードを同一ダイアログへ同居させ、コードを転記させることで「ユーザーが
action ラベルを見ざるを得ない」状態を作る。レビュー案より攻撃面が小さく、目的（悪化させない
＋疲労緩和）を満たしており、こちらのほうが優れた判断である。

## 新規の指摘（重要度順）

### N1（最重要・内部矛盾＋仕様の穴）: iPhone へのチャレンジ出所認証を「セッション認証」に代替させない

**問題**
§9.5 は iPhone 側の検証項目を「Linux 側署名**または**セッション認証が正しい」、§8.2 は保存物を
「Linux verifier 公開鍵**または**証明書」と、いずれも "または" で二択未確定のまま残している。
これは §10.1 と衝突する。§10.1 で**暗号化チャネルは非特権 `faceid-transport` で終端する**と
決めた以上、iPhone がチャネルの「セッション認証」で相手を認証すると、それは verifier ではなく
**チャネルを終端している helper を認証してしまう**。verifier の秘密鍵は root 側にあり helper は
持てないため、チャネルを verifier 鍵で相互認証すること自体できない。

さらにこれは仕様の穴でもある。§9.5 は「Linux 側署名」を前提にしているのに、§9.1／§9.4 は
**iPhone が署名する対象**しか定義しておらず、**verifier が送出チャレンジに付す署名**（どの鍵で・
どのバイト列に）が定義されていない。指摘 A で `verifier_ephemeral_key` を削除した結果、この
出所認証の担い手が宙に浮いている。

**推奨対応**
- チャレンジ出所の認証は、チャネルとは独立した、正規化済みチャレンジに対する **verifier の
  エンドツーエンド署名一択**に確定する。「セッション認証」という代替は削除する。
- §8.2 でペアリング時に iPhone が保存する verifier 公開鍵を用い、canonical challenge 全体に
  verifier が署名することを §9 に明記する（署名鍵・対象バイト列・アルゴリズム）。

**実害の正確な範囲**
これが欠けても、偽造チャレンジで得た署名は verifier 側で request_id 非保留として弾かれるため、
**認証の突破には直結しない**。効くのは、helper が iPhone に任意の偽コンテキストを表示させる／
通知を連打する経路（ユーザーを騙す・DoS）を塞ぐ点。「通信路だけを信用しない」という本設計の
原則を貫くうえで、E2E 署名に確定させるべき。

### N2（number matching の実装依存リスク）: `PAM_TEXT_INFO` が実際に表示される保証

**問題**
§9.3・§11.1 の順序制御（verifier がコードを PAM へ返す → PAM が `PAM_TEXT_INFO` で表示 →
表示成功を verifier へ通知 → 通知後に challenge を helper へ渡す）は思想として正しいが、
二点の確認・明記が要る。

1. **`conv()` の `PAM_SUCCESS` ＝ ユーザーがコードを見た、ではない。** polkit エージェント
   （polkit-gnome / KDE 等）は PAM info メッセージを表示するが、パスワードプロンプトを伴わない
   info 単独メッセージの描画挙動はエージェント依存。§11.1 の「表示成功通知」は「conv が成功を
   返した」以上を保証しない。
2. number matching の効き目は「コードと action ラベルが**同一ダイアログに同居**する」ことに
   依存している。

**推奨対応**
- 対象エージェントで info 単独メッセージが認証開始前に確実に描画されることを実機確認する。
  描画できないエージェントでは §9.3 のフォールバック（Face ID を開始しない）に確実に落とす。
- 「エージェントは action メッセージと PAM info（コード）を同一ダイアログに並べる」という前提を
  §9.3 か §13.3 に明示する。

### N3（小さいが重要な不変条件）: iPhone はチャレンジ内の確認コードを表示・自動入力しない

**問題**
確認コードは §9.1 で署名対象に含まれ、チャレンジとして iPhone にも届く。iPhone がこれを画面に
表示したり入力欄へ自動補完したりすると、ユーザーは Linux 画面を見ずに iPhone だけで完結でき、
number matching の反射承認対策がゼロになる。§9.5 の表示例（「Linux 側に表示されたコードを入力」）は
意図として正しいが、不変条件として明記されていない。

**推奨対応**
- 「iPhone は受信した確認コードを表示・自動補完せず、ユーザー入力との**照合にのみ**使用する」
  という不変条件を §9.3 か §13.3 に一行明記する。

## その他（軽微）

- **iPhone 側の用途許可リストの配布**: §9.5 で iPhone が「PAM サービスから導出した用途が許可
  対象である」を検証するが、その許可リストをどう配布・更新するか（チャレンジ経由で攻撃者に
  書き換えられないこと）が未定義。ペアリング時に固定する等を §8 か §21 に追記する。
- **リカバリーコード**: §14.4 の選択肢として挙がっているが、採用するなら保存先と検証経路
  （別の PAM ファクターであること）の定義が要る。

---

## 補足: 「PAM サービスから導出した用途」の意味（レビュー観点の共有用）

`requested_action`（§9.1）は「この認証が何のための認証か」を表す。設計はこれを、呼び出し元が
root コンテキストで設定し途中で改竄されない **PAM サービス名**（`PAM_SERVICE`）から機械的に
導出する（`polkit-1` → `polkit-1.authenticate`、`sudo` → `sudo.authenticate` 等）。helper の申告・
モジュール引数・会話表示文字列は出所を保証できないため用途源にしない（§9.2・§12.4）。

- **防げる**: ある PAM サービス用の署名を別 PAM サービスへ流用する攻撃（polkit ↔ sudo ↔ login）。
- **防げない**: polkit 経由は一律 `polkit-1` に潰れるため、1Password 解錠と `pkexec` を用途で
  区別できない（action 跨ぎ）。これが「polkit は action ID を PAM へ渡さない」制約の
  `requested_action` から見た姿であり、§4.1／§4.2 の防御範囲限定と整合している。

## 総合と優先順位

v0.3.0 は実装フェーズに入れる品質で、A〜E と C はきれいに閉じている。

1. **N1（必須・ワイヤプロトコル確定前）**: 送出チャレンジへの verifier エンドツーエンド署名を
   §9 に定義し、「セッション認証」代替を削除する。§10.1 との整合として必須。
2. **N2**: number matching の `PAM_TEXT_INFO` 表示を対象エージェントで実機確認し、同一ダイアログ
   同居の前提と表示失敗時フォールバックを明記する。
3. **N3**: 「iPhone は受信コードを表示・自動補完しない」不変条件を明記する。
4. **軽微**: iPhone 側許可リストの配布方法、リカバリーコードの仕様。

N1 のみがプロトコル設計上の未クローズ点。N2・N3 は新設 number matching 機能の堅牢化（実装時
確認と不変条件の明記）。残りは記述の仕上げ。
