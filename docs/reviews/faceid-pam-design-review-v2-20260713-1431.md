## 総評

前回の指摘 7 件は、すべて設計書全体で整合を取った形で反映されている。一箇所だけ直して他が
矛盾する、という直し方をしておらず、対応の質が高い。特に polkit の用途束縛（前回指摘 1）は、
脅威モデル §4.1 / 非ゴール §4.2 / プロトコル §9.2 / iPhone 表示 §9.4 / PAM 統合 §12.4 /
要約 §24 / スコープ §22 / 未決事項 §21 まで一貫して波及している。§26 の変更履歴も適切。

v0.2.0 は実装フェーズに入れる品質。着手前に潰すべきは実質 1 点（下記 A）のみ。

なお本レビューは「本設計が導入した問題」と「polkit などの基盤に内在する既存制約」を区別して
扱う。特に指摘 C は後者に属し、設計の責務は問題を"解決"することではなく"悪化させないこと"に
ある。この切り分けを曖昧にすると、内在制約をあたかも本設計の欠陥のように誤読させてしまう。

## 前回指摘への対応確認（すべて解決）

- **指摘 1（polkit の用途束縛）**: §9.2 で `requested_action` を `PAM_SERVICE` 由来に変更。
  polkit が action ID を PAM へ渡さないこと、rules.d では PAM モジュールを action 別に
  選べないこと、許容できなければ polkit 統合せず専用サービスまでに留めることを明記。
  §4.1 の脅威を「別 PAM サービスへの流用」に正しく限定し、§4.2 に「polkit-1 内の
  action 間分離はしない」を非ゴールとして追加。→ 解決。
- **指摘 2（ペアリング SAS）**: §8.2 で比較コードを「両公開鍵＋両 device ID＋ペアリング
  nonce＋バージョンを含む正規化トランスクリプトのハッシュ由来の SAS」と規定。
  iPhone→Linux の戻り公開鍵も束縛対象に。→ 解決。
- **指摘 3（monotonic TTL）**: §9.6-4 および §9 末尾で TTL の権威を verifier の
  monotonic clock に。§18.3 に wall-clock 変更耐性テストも追加。→ 解決。
- **指摘 4（チャネル終端点）**: §6 図と §10.1 でチャネルは helper 終端、完全性は署名連鎖
  由来、チャネルは defense-in-depth と明記。→ 解決。
- **指摘 5（fail closed の用語）**: §19 で当該語を廃し、「エラー・明示拒否でも
  `PAM_SUCCESS` を返さずフォールバック」と明確化。→ 解決。
- **指摘 6（polkit-1 の .pacsave 退避）**: §19 に更新後ヘルスチェックと
  `.pacsave` / `.pacnew` 検出を追加。→ 解決。
- **指摘 7（細かい点）**: §13.1 のフラグ整理、§8.1 の belt-and-suspenders 明記、
  §9.6 のアトミック consume、§12.1 の `pam_sm_acct_mgmt` no-op、すべて反映。→ 解決。

## 新規・残存の指摘（重要度順）

### A（唯一の実質的な穴・着手前に要確定）: `verifier_ephemeral_key` の役割が未定義で §10.1 と衝突

**問題**
§9.1 の署名対象に `verifier_ephemeral_key` が含まれるが、この項目の意味と用途が本文の
どこにも記述されていない。名称からは「verifier が張るチャネルの一時鍵に署名応答を束縛する
（channel binding）」意図に読めるが、§10.1 で**チャネルは helper 終端**と決めたため、
verifier はチャネルの端点ではなく、この鍵が何にバインドするのかが宙に浮く。

関連して §9.6 ステップ 10「応答が同一セッションへ束縛されている」も、"セッション" の具体が
未定義。チャネルが helper 終端かつ untrusted である以上、セッション束縛をトランスポート層に
依存させると信頼境界の外へ足を置くことになる。実際には request_id + nonce + 署名で束縛は
足りているため、`verifier_ephemeral_key` と "セッション束縛" は次のいずれかである。

1. request_id 束縛と重複しているだけ
2. 本来やりたかった verifier↔iPhone の end-to-end channel binding が helper 終端で崩れている

**推奨対応（二択）**
- 冗長なら `verifier_ephemeral_key` を削除し、束縛は request_id / nonce / monotonic-TTL に
  一本化する。
- E2E channel binding を本当に行うなら、helper 終端のトランスポートの上に verifier↔iPhone の
  アプリ層鍵合意を一段重ね、その公開鍵を署名チャレンジに含める設計を明記する。この場合、
  §10.1 の「チャネルは helper 終端」と両立する書き方にする。

プロトコルのバイト列（canonical encoding）を確定する前に、この 1 項目の意味を決めること。
本設計で唯一の未クローズなプロトコル設計点。

### B（具体的な実装指定）: TTL は `CLOCK_MONOTONIC` ではなく `CLOCK_BOOTTIME` を使う

**問題**
§9 末尾で monotonic clock に寄せた判断は正しいが、Linux の `CLOCK_MONOTONIC` は
**サスペンド中は進まない**。認証保留中に端末がスリープし、長時間後に復帰した場合、
30 秒 TTL が実時間で大幅超過しても monotonic 上は数秒しか経過しておらず、期限判定が
甘くなる抜けが生じ得る。

**推奨対応**
TTL のようなセキュリティ用途では、サスペンド時間も算入する `CLOCK_BOOTTIME` を指定する。
§9.6-4 / §9 末尾 / §16 に「TTL 判定は `CLOCK_BOOTTIME` 基準」と明記する。

### C（脅威モデルの位置づけ明確化）: 要求取り違え／承認疲れは本設計が導入した穴ではなく、polkit 内在の制約

**この項目の意図（誤読の防止）**
現行 §4.1 は「認証対象を誤認させる承認疲れ・要求取り違え」を無条件の防御対象として挙げており、
読み手には「本設計が新たに開けた脆弱性」と受け取られかねない。実際は逆で、これは polkit の
認可モデルに内在する制約であり、既定のパスワードフローにも同じく存在する。「今クリックした
自分の操作がこの認証要求を発火させた」という因果は、認証方式が何であれ polkit 側では検証
できない（confused-deputy とタイミング一致の問題）。したがってこの指摘の意図は「設計が穴を
作った」という非難ではなく、下記の切り分けを設計書に明記させ、設計の責務を正しく限定すること
にある。

**本設計が導入したわけではない（既存・不可避）**
- 因果の非検証性は polkit 内在。`pam_unix` の既定フローでも、`pam_fprintd` で 1Password を
  解錠する構成でも等しく成立する。本設計がこの脆弱性クラスを生み出しているわけではない。

**本設計が"悪化させ得る"二点（ここが設計の責務）**
1. **intent ラベルの消失**: データフロー上、`action_id` と `message` は polkitd → 認証エージェント
   までは届くが、エージェントが起動する helper 経由の PAM には `service=polkit-1` しか渡らない。
   このため PAM モジュール単独実装だと、既定エージェントが表示していた「ネットワーク設定を
   変更…」という唯一の識別シグナルを消してしまう。**pkexec と正規操作が本当に区別不能になるのは
   既定フローではなく本設計側**である。
2. **摩擦低下による疲労増幅**: パスワード入力という手間自体が注意のチェックポイントだった。
   Face ID / 押し通知はそれを除くため、反射承認が増えやすい。

**推奨対応**
- §4.1 の当該項目を「polkit 内在の制約であり本設計が導入したものではない」と位置づけ直し、
  以下 (a)〜(c) を設計の責務として記す。
- (a) intent ラベルを消さないため、**PAM モジュール単独ではなく polkit 認証エージェントとして
  実装**し、`action_id` / `message` / `details` を捕捉して iPhone に中継する（既定フローと同等
  以上の透明性を確保。subject まで見せれば既定を上回れる）。文脈中継は PAM 層では原理的に
  手遅れでエージェント層でしかできない、という点も明記する。
- (b) 摩擦低下は number-matching 型（要求元に表示したコードを iPhone に入力させる）で相殺する。
- (c) 残る因果非検証は rules.d で高影響 action を faceid 経路から外して爆発半径を絞り、
  受容する旨をスコープに明記する。

### D（軽微）: テスト設定のモジュール引数が §9.2 / §12.4 の決定と不整合

**問題**
§12.4 で「モジュール引数・環境変数・helper 申告で用途を渡す方式は採用しない」と決めた一方、
§18.4 のテスト設定には `action=test.authenticate` というモジュール引数が残っている。専用
テストサービスの単なるラベルなら害はないが、「用途は `PAM_SERVICE` から導出」という一本化
方針と表面上ぶつかる。

**推奨対応**
テストサービスも `PAM_SERVICE`（例: `faceid-auth-test.authenticate`）から導出に統一するか、
「モジュール引数はセキュリティ境界ではなくテスト用ラベルに限る」と明記して整合を取る。

### E（軽微）: nonce のエントロピーも明記する

**問題**
§16 で request_id を 128bit 以上と規定した一方、nonce の長さ・生成源が未指定。

**推奨対応**
nonce も「128bit 以上の CSPRNG」と明記して揃える。

## 懸念なしと判断してよい項目（参考）

- **§21.9「verifier 再起動時の保留 request 破棄」**: セキュリティ的には既に安全側。
  再起動で保留状態を失えば、後から届いた応答は §9.6-1「保留中である」で弾かれ、
  fail-closed になる。これは実装詳細であって脆弱性ではない。
- **ECDSA の署名 malleability**: リプレイの鍵を署名ではなく request_id に置いている限り無害。

## 着手前の優先順位

1. **A**: `verifier_ephemeral_key` とセッション束縛の意味を確定する（canonical encoding 確定前に必須）。
2. **B**: TTL を `CLOCK_BOOTTIME` 基準と明記する。
3. **C**: 要求取り違えを「polkit 内在の制約であり本設計が導入したものではない」と位置づけ直し、
   intent ラベル維持のためエージェント実装を採るか判断する（記述整合に加えアーキテクチャ選択を含む）。
4. **D / E**: テスト設定・nonce 記述の整合仕上げ。

A がプロトコル設計上の唯一の未クローズ点。C は記述整合にとどまらずアーキテクチャ選択
（エージェント実装の是非）を含むが、着手可否を止めるものではない。B / D / E は実装時の指定・
記述整合。これらを反映すれば §23 の実装順序にそのまま乗れる品質。
