## 第 1 章 Linux の最小モデル

1. **一次的な根拠は UID（数値）である。**
   ユーザー名は人間のための見た目で、カーネルは名前を UID に対応づけてから権限を判断する（§1.1）。
   名前と UID の対応は `/etc/passwd` などが持ち、その引き当ての共通機構が NSS である。

2. **root は本書のパーミッションモデル（DAC）の中で非常に強い権限を持ち、通常のパーミッション判定を
   ほぼ素通りして他ユーザーのファイルの読み書き・任意プロセスの停止・システム設定の変更ができる。**
   一方で「何でもできる」とは言い切れない。理由の例: capability、LSM（SELinux/AppArmor）、名前空間、
   ファイルの immutable 属性といった追加機構が root の操作を制限する場合があるため（§1.1）。

3. **(a) 上がらない。** IPC で依頼しても、それは通信をしただけで、依頼した `alice` のプロセスは
   `alice` の資格情報のままである。root のサービスは別のプロセスであり、要求を検証し実行するのは
   その root プロセスが自分の資格情報で行う。要求を投げる側と判断する側ははじめから別プロセスで、
   IPC は権限の昇格ではない（§1.2）。
   **(b)** 同じログイン操作からでも、`setuid` された実行ファイル、`sudo`、systemd によるサービス起動
   など、あらかじめ管理者が許した認可済みの経路を通ったときだけ、`alice` と異なる資格情報（実効
   UID `0` など）のプロセスが生まれる（§1.2）。通常のシェルは `alice`、`sudo` 経由で生成された別
   プロセスは実効 UID `0`、というように経路で資格情報が変わる。これは勝手な昇格ではなく、管理者が
   明示的に用意した扉である。

4. **読み取りも書き込みもできない（実行もできない）。**
   `0640` は「owner=読み書き、group=読み取り、other=不可」を表す。所有者は root、所有グループは root で、
   対象ユーザーは root グループに属さず ACL もないという条件なので、そのユーザーは other 区分に該当し、
   read も write も与えられていない（§1.3）。
   注意: この結論は「所有グループのメンバーでない」「ACL なし」という条件があって初めて確定する。
   もし対象ユーザーが所有グループのメンバーなら、group 区分の read が効いて読み取りは可能になる。
   （root 自身や、権限を迂回する capability を持つプロセスは本問の対象外。）

5. **認証の最終判断と、その判断が依存するデータ（登録済み鍵など）を、root 側（root 所有かつ一般ユーザー
   書き込み不可のパス）に置くべきである。**
   理由: 一般ユーザーのセッションは侵害されうる（＝そこで動くコードやデータは改ざんされうる）が、
   root 側は侵害されていない前提で信頼する土台（TCB）として扱う。判断ロジックと依存データを信頼できる
   側に置けば、非特権マルウェアは要求を出すことしかできず、その中身は root 側で検証される（§1.5）。
   （正本 §3「root 攻撃者は信頼しない」／§4.2「root 取得済み攻撃者は防御対象外」は、「root がすでに
   乗っ取られた状況は守備範囲の外」という意味であり、root 側を TCB として扱うことと矛盾しない。）

6. **ファイル自身の mode を `root:root 0640` にしても、親ディレクトリを非特権者が書けるなら、
   元のファイルを削除（unlink）・改名（rename）して攻撃者の用意した別ファイルへ置き換えられるため、
   内容の改ざんを防げない。**
   守るべきはファイル一つではなく、そのファイルへ至るパスを構成するディレクトリすべてであり、
   加えて開くときに symlink のすり替えや TOCTOU を避ける安全な手順が要る（§1.3、§1.5、正本 §8.2）。

7. **systemd はサービス本体へ「待ち受けソケット（listening socket）のファイルディスクリプタ」を渡し、
   接続を accept するのはサービス本体（bifrauth の中核デーモン）である。**
   サービスが起動するまでの接続は、カーネルの待ち行列（backlog）で待たされる（§1.4）。
   （別方式として、接続ごとに systemd 側が accept 済みの接続を渡す設定もあるが、bifrauth は使わない。）

## 第 2 章 認証と認可、脅威モデルの立て方

1. **認証は「あなたは誰か」を確かめること、認可は「ある操作を許すか」を決めることである。**
   典型的には認証結果が認可の入力になるが、常に認証が先とは限らない。認可は相手が誰か分からないまま
   下されることもある（例:「誰でも入ってよいロビー」は本人確認なしに立ち入りを許す認可）（§2.1）。

2. **Face ID そのものは Linux の `alice` や要求元プロセスを識別しない。** 正しくは、`alice` の uid に
   登録された端末の鍵が、root 側 verifier が作ったその場かぎりの要求（誰・どのサービス・どの端末・
   期限に束縛）へ署名し、その署名が検証を通ることが本人性を支える。Face ID はその署名に使う Secure
   Enclave 鍵の使用を許可するゲートにすぎない（§2.1）。

3. **汎用の高速ハッシュ一回では、辞書攻撃や総当たりで大量に試すのが容易だから。** ユーザーごとの
   ソルト（同じパスワードでも保存値を変える）と、計算を重くするコストを組み込んだ専用の変換（パスワード
   ハッシュ/KDF、Linux なら shadow の方式）を使う。それでも保存値が漏れれば手元での候補試行（オフライン
   推測）は可能で、弱いパスワードは破られうる。だから「復元しにくい」だけで安全とは言えない（§2.2）。

4. **なりすましは、攻撃者が正規ユーザーとして扱われてしまう結果。リプレイは、過去の正しいやり取りを
   そのまま再送する手法。** 一方は結果、一方は手法であり、必ず同時に起きるわけではない。
   **(a) 同一文脈での過去要求の再送を防ぐのは「新しさ」（freshness、一回きりの nonce・期限・使用済み
   記録）。 (b) 別のサービスや端末への流用を防ぐのは「文脈への束縛」（context / audience binding）。**
   署名で「正しい相手が作った裏付け」（authenticity）を確かめても、それだけではリプレイを防げない。
   正しい署名つきの古い要求がそのまま再送されうるので、新しさの検査が別途必要である。三つの性質は
   互いを肩代わりしない（§2.3）。

5. **一つめ（リプレイ）: `true` を盗んで再送すれば通る。二つめ（偽造 forge）: 盗む必要すらなく、攻撃者が
   自分で `true` を作って送れる。** 危ういのはブール型だからではなく、成功したという主張だけを、鍵に
   よる証明もその場かぎりの値も宛先・用途への束縛もないまま信じてしまうことである（署名付き・一回性・
   宛先束縛のトークンなら主張を裏付けられる）。bifrauth は真偽値を信頼せず、Secure Enclave 鍵による
   その場かぎりの要求への署名を検証する（§2.3、正本 §5.1）。

6. **例: 非ゴール=初期版で遠隔認証を提供しない。前提=Secure Enclave の鍵は取り出せない（正本 §3）。
   範囲外の脅威=root をすでに奪った攻撃者（正本 §4.2）。残余リスク=number matching で緩和しても、
   ユーザーが確認を怠れば残る要求の取り違え。** これらを区別せず「守らない」で一括りにしないことが
   要点（§2.4）。
   **「ユーザーの誤承認」と「DoS」は、いずれも正本 §4.2 が明示的に防御対象外に挙げる「範囲外の脅威」**
   であり、残余リスクではない。分類は「何を守ると宣言したか」で決まり、レート制限などを多層防御で
   実装しても分類は変わらない（§2.4）。

7. **bifrauth では判断を下す verifier・PAM モジュール・登録情報が信頼境界の内側（TCB）にある。**
   root を奪った攻撃者はそれらを書き換えられるので、この設計が与える「Linux 側の認証成功判定」の保証は
   そもそも成り立たなくなる。範囲外に置くのは能力不足ではなく、約束の境界を宣言することである
   （§2.4、正本 §4.2）。「どんな認証も意味を失う」という普遍的な断定ではない点に注意。

8. **鍵の在りか: 偽造に必要な鍵は Secure Enclave の中にあり取り出せないと仮定する（正本 §3）。判断の
   場所: 認証の判断は root 側（TCB）が下す。送られる証の性質: その場かぎりで使い回せない。** この三つ
   により、非特権側が丸ごと侵害されてもそのプロセス単独では偽造できない。ただし無条件ではない。攻撃者は
   正規要求の誘発、承認疲れ・要求取り違えの誘導、DoS などが可能で、root まで奪われれば保証は成り立た
   ない（§2.5）。

## 第 3 章 PAM

1. **各アプリケーションが「本人確認をどうやるか」を自前で書く必要がなくなる。** PAM がなければ、認証
   方式を一つ足すだけで、`login`・`sudo`・画面ロック・SSH などすべてを書き換えて回ることになる。PAM が
   あれば、確かめ方をモジュール（`.so`）として差し込むだけで、アプリを書き換えずに足せる（§3.1）。

2. **サービス名はアプリ起点の値である。** カーネルが実行ファイルを検証して付ける値ではなく、PAM を呼ぶ
   アプリケーションが開始時に指定し、モジュールは `PAM_SERVICE` として受け取る（§3.1）。bifrauth が
   モジュール引数や非特権 helper の自己申告ではなく `PAM_SERVICE` を用途の根拠にするのは、信頼できる
   呼び出し元（root 専用 IPC と `SO_PEERCRED` で確認）が渡す値だからである。ただしサービス名だけでは、
   実際にどの実行ファイルやどの操作（polkit の action など）から来たかを暗号学的には証明できない
   （特に polkit は action ID を PAM へ伝えない。第 4 章）。

3. **アプリケーション: PAM に本人確認を依頼する側（`login`・`sudo`・`polkit-agent-helper-1`）。モジュール:
   実際に確かめる処理を行う `.so`。conversation: モジュールがユーザーと言葉をやり取りするための通り道。**
   モジュールは画面へ直接は描けず、conversation を通じてアプリケーションに表示や入力を頼む（§3.2）。

4. **メッセージが実際にユーザーの目に触れたことは保証できない（「表示できた ≠ 見られた」）。** bifrauth の
   number matching で確認コードを表示するとき、この隙間が問題になる（§3.2、第 20 章）。

5. **auth は「あなたは誰か」（本人確認）に、account は「その本人に今このアクセスを認めてよいか」（期限・
   時間帯・端末などの条件）に答える。** bifrauth が主に住むのは auth。account に対しては、状態を変えず
   成功を返す no-op として振る舞う（§3.3、正本 §12.1）。**この no-op は「account スタック全体を許可する」
   意味ではない。** 「bifrauth のモジュールが account の観点で追加の拒否をしない」だけで、account を通すかは
   スタックに並ぶ他のモジュールとフラグが決める。

6. **`PAM_AUTH_ERR` は本人確認が明示的に失敗した状態、`PAM_AUTHINFO_UNAVAIL` は確認情報が得られず確認
   そのものを行えなかった状態を指す。** 単純な `sufficient` 構成では両者とも下段のパスワードへ落ちるが、
   落ちる先が同じでも意味は違う。アプリケーションの診断、監査ログ、ユーザーへの案内、`[value=action ...]`
   を使った細かいポリシーでは両者を別々に扱える。（ただし失敗の詳細をそのままユーザーへ見せることが常に
   良いとは限らない。攻撃者への手がかりになりうるため、どこまで見せるかはアプリ側の判断）（§3.5）。

7. **`required` は失敗しても即座には止めず、スタックの残りを評価してから最終結論を失敗にする（どの
   モジュールで失敗したかを悟らせない）。`requisite` は失敗した瞬間に打ち切り、残りを評価しない**（§3.4）。

8. **それより上に置かれた `required` なモジュールがすでに失敗している場合は、`sufficient` が成功しても
   打ち切れない。** スタックは上から下へ順に評価され、順番と組み合わせで結論が変わる（§3.4）。

9. **認証全体はパスワード認証（`common-auth`）の結果で決まる。** iPhone 未到達による `pam_bifrauth.so` の
   失敗は `sufficient` なので無視され、評価は二行めのパスワードへ落ちる（§3.6）。ただしこれが成り立つ前提:
   (a) モジュールが期限内にきちんと非 `PAM_SUCCESS` を返して戻ること（戻らず/呼び出し元を巻き込んで落ちない
   こと、panic を封じ込めること）、(b) 下段のスタックが正しく設定されていること（`common-auth` はディスト
   リビューション定義で、その下段自身の結果で可否が決まる。インストーラーは下段を勝手に上書きしない）。

10. **成功でないものを `PAM_SUCCESS` と偽ると、`sufficient` がそこで認証を通してしまい、下段のパスワードへ
    落ちる道が塞がれるうえ、本来通らないはずの認証が通ってしまうから**（§3.5、§3.6、正本 §19）。
    **`sufficient` が無視するのは「返された非成功」なので、モジュールが結果を返さず呼び出し元プロセスごと
    落ちてしまった場合は救えない。** だからモジュールは期限を設け、panic を呼び出し元へ伝播させないよう
    封じ込める（第 11 章・第 22 章）。

## 第 4 章 polkit と 1Password の system authentication

1. **polkit は主に認可を引き受ける。** 「ある主体（subject、要求してきたプロセス）が、ある操作を実行して
   よいか」をシステム全体で判断する調停役である（§4.1）。本人確認そのものは polkit の仕事ではなく、
   必要なときは PAM へ委ねる。

2. **action は polkit が扱う一つひとつの操作で、action ID で識別される。** 既定の結果の例: 無条件に許す、
   無条件に拒否する、本人の認証を求める（auth_self）、管理者の認証を求める（auth_admin）（§4.1）。
   **auth_self が認証するのは「要求してきたプロセスの Unix ユーザーそのもの」とは言い切れない。** セッション
   の持ち主にあたる identity を認証するのであり、主体（subject）と認証される identity は区別される。

3. **同じ action ID と主体の組に一時的な認可（temporary authorization）が保持される（keep）と、その間の
   判定は認証エージェントや PAM を再び呼ばずに「許可」になることがあるから。** これは polkit が認可の結果を
   しばらく保持する性質であって（単に PAM の成功結果をキャッシュするのとは違う）、bifrauth の設計上の保証や
   UX とは別物。混同してはいけない（§4.1）。

4. **認証エージェント（authentication agent）がユーザーへ尋ねる役を担い、ユーザーのセッションの中＝
   ユーザー権限で動く。** 実際の判定は特権を持つヘルパー（対象環境では `polkit-agent-helper-1`）が担う。
   **ただし、どのヘルパーを経てどの PAM スタックを通るかは、エージェント実装・polkit の版・ディストリ
   ビューションに依存する実装上の経路であり、常に同じとは断定できない。** 実際に `polkit-1` の PAM スタックを
   通るかも実機確認の対象になる（§4.2）。

5. **action ID は認証エージェントには届くが、PAM モジュールや verifier には届かない。** ヘルパーが PAM を
   サービス `polkit-1` で起動する境界で action ID は引き継がれない。だからエージェントは action の説明
   （intent 表示）を出せるが、PAM から先はどの action か分からない。表示は非特権エージェントによるもので
   PAM へも届かないため、その表示内容を暗号学的な判定や署名対象には使えない（§4.2、§4.3、正本 §9.3）。

6. **(a) 非特権エージェントが実行時に申告する場合: その申告者自身を認証できないから採れない。**
   **(b) root 管理の PAM 設定へ静的に `1password.unlock` と書く場合: 攻撃者の自己申告ではないが、同じ
   `polkit-1` スタックを通るすべての action を一律に「1Password だ」と誤ってラベル付けするだけで、実際に
   どの action から来たかを証明したことにはならない。** どちらも action ID の証明にならず、設計はこの
   用途申告方式を採らない（§4.3、正本 §12.4）。

7. **できること: `rules.d` は action ID ごとに許す/拒否する/本人認証/管理者認証を返せるので、ある action
   だけ認証を要求し他を許可/拒否する制御はできる。できないこと: 認証を要求する複数の action の間で、
   action ごとに別の PAM サービス・スタック・モジュールを選ぶことはできない。** auth_admin にしても、
   認証される identity が管理者になるだけで別の PAM スタックへは移らず、依然 `polkit-1` を共有する
   （§4.3、正本 §12.4）。

8. **bifrauth を入れた `polkit-1` の認証スタックを、他にどの polkit action が共有しているかを必ず確かめる
   こと。** 同じスタックを通る action には Face ID が使えるようになるため、その範囲を把握する（§4.3、
   正本 §12.4）。

9. **意味しない。** 正本 §3 は、その表示を「満たされているべき前提（受入条件）」として記述しているだけで、
   「すでに確認した」という報告ではない。実際、対象エージェント/版でのこの確認はまだ行われておらず
   （progress.md 項目 14 は未着手）、現時点では未検証である。設計は、この確認が取れるまで polkit 統合を
   有効化・サポートしない立場を取る（§4.4、正本 §9.3 / §19）。未確認のことを保証に数えない。

## 第 5 章 暗号の最小限と Secure Enclave

1. **検証単独で分かるのは、(1) その公開鍵に対応する秘密鍵で作られた有効な署名であること、(2) 署名対象が
   署名後に変えられていないこと、の二つだけ。** 「その公開鍵が確かにあの iPhone のものだ」という結び付き
   （誰の鍵か）は検証ではなく、安全なペアリングとレジストリ（第 10 章・第 18 章）が与える（§5.1）。

2. **署名は「中身を隠す」暗号化とは別の操作で、示すのは「対象が秘密鍵と結び付いた形で作られ、署名後に
   変えられていない」ことだから。** また、攻撃者が登録済みの公開鍵を自分の鍵へ差し替えられた場合、
   攻撃者の署名も「有効」として通るので、検証が通ること＝本人性、とは言えなくなる。検証の結論は、
   公開鍵と端末の結び付き・秘密鍵非漏洩・方式が破られていないこと、という前提の上に立つ（§5.1）。

3. **耐性: 原像計算困難性・第二原像計算困難性・衝突困難性（＋設計上のなだれ効果）。** 256 ビット出力
   では異なる入力が同じ値になる組（衝突）は理論上必ず存在するので「衝突は存在しない」とは言えない。
   暗号学的ハッシュが与えるのは「衝突を現実的に見つけられない」こと。ハッシュ一致は同一性の証明ではなく、
   その前提の下での比較である（「現実的に困難」≠「不可能」）（§5.2）。

4. **(1) 自分が今発行して保留（pending）に持つ nonce/request ID と一致するか確かめる、(2) 有効期限を
   過ぎていないか確かめる、(3) 一度使った要求をその場で一度だけ消費（consume）し二度と通さない。**
   nonce がメッセージにあるだけ・毎回乱数を作るだけでは古い答えを拒否できない。新しさを保証するのは
   検証する側の状態管理である（§5.3。TTL の権威は CLOCK_BOOTTIME の保留状態。第 8・16 章）。

5. **別の文脈への流用を防げる。** 署名対象に用途（PAM サービス由来）が含まれるので、`polkit-1` 向けの
   答えを `sudo` 向けに使おうとしても、署名対象の用途が違い検証が落ちる。端末 A 向けを端末 B へ流しても
   同様に落ちる（§5.4。第 2 章の「文脈への束縛」の正体が、署名対象へ文脈を含めること）。

6. **署名対象は応答メッセージ一式ではなく、保存済みの canonical challenge のバイト列全体。** 応答の
   request ID・iPhone 端末 ID・`signed_payload_hash` をまとめて署名しているのではない。`signed_payload_hash`
   （=SHA-256(canonical_challenge)）は照合を助ける補助値にすぎず認証入力として信用せず、verifier は保留中の
   canonical bytes から自分で計算し直して照合し、署名検証もその canonical bytes に対して行う。iPhone 端末 ID
   はどの登録鍵で検証するかのヒントで、本人性の根拠は「選んだ登録鍵での署名検証が通ったこと」（§5.5）。

7. **前提: Secure Enclave の秘密鍵は原則として外へ取り出せない、と扱う（第 2 章の assumption。正本 §3。
   iPhone OS や SE の完全な侵害は out-of-scope＝正本 §4.2 の範囲外）。** 「Face ID でしか使えない鍵」は
   無条件ではなく、鍵を SE 内に署名専用で作る・`.biometryCurrentSet`・署名ごとの新しい認証コンテキスト・
   バックグラウンド後の無効化などが正しく実装・運用される条件つきで成り立つ。実機 iPhone は未実装
   （mock 代替）である点も忘れない（§5.6）。

8. **Face ID は「認証の結果」ではなく、Secure Enclave の秘密鍵を「今使ってよい」と許可するゲートとして
   使われる。** Linux 側が最後に検証するのは「Face ID が成功したという申告」ではなく署名である。Face ID の
   成功は鍵の使用を認可するだけで、ユーザーが何を承認したつもりかという意図も、Linux 側のどの action かも
   証明しない（§5.6。反射的承認は number matching で補う。第 20 章）。

9. **欠けている二つの性質: (1) 偽造できない性質（裸の主張は秘密鍵なしに誰でも作れる）、(2) 使い回せない
   性質（一度通った主張を再送できる）。** 署名は、その場かぎりの nonce を含み文脈に束縛された問いかけへの
   署名なので、(1) 作るには Face ID でしか使えない秘密鍵が要り偽造できず、(2) 検証側が保留中の問いかけとの
   一致・期限・一度きりの消費を確かめるため再送もできない。**危ういのはブール型だからではなく、裏付けの
   ない裸の主張を信じることにある**（署名・MAC 付きで nonce/宛先に束縛したトークンなら一般には設計可能）
   （§5.7）。

10. **署名が保証するのは、問いかけと答えの出所・完全性を両端で確かめられること（end-to-end）だけである。**
    署名は通信の中身を隠すこと（機密性）も、通信量の観察や妨害（DoS、正規メッセージの握りつぶし）を防ぐ
    ことも、しない。盗聴防止や非ペア端末の抑止は相互認証された暗号化チャネルという別の層が担うが、方式は
    未決で実装もこれから（第 10 章・第 23 章）（§5.5）。

11. **渡すのは canonical challenge の生のバイト列全体である（ハッシュではない）。** メッセージ用 API は
    内部で SHA-256 をちょうど一度だけ行う。誤ってあらかじめ計算した 32 バイトのハッシュを渡すと、その
    ハッシュにさらに SHA-256 が掛かって二重ハッシュになり、Linux 側が canonical bytes に対して行う検証と
    対象がずれて、署名検証が通らなくなる（§5.2。詳細は第 14 章）。

## 第 6 章 ローカル通信とファイル・永続化の安全性

1. **Unix domain socket は同じマシンの中のプロセス同士をつなぐ通信路で、多くはファイルシステム上のパスを
   名前に持つ。** bifrauth は PAM モジュールと verifier の通信に `/run/bifrauthd/pam.sock` を使う（§6.1）。

2. **パーミッションだけでは、設定ミスや環境ごとの挙動の違いに頼ることになるから。** 実際につないできた
   相手が誰かを OS に直接尋ねる `SO_PEERCRED` を重ね、多層で確かめる（§6.1、§6.2、正本 §11.1）。

3. **カーネルが埋める。** 相手が自分で「root です」と名乗るのではなく、接続成立時の相手の資格情報を
   カーネルが記録して返すため、非特権プロセスがこの値を偽って root のふりをすることはできない（§6.2）。

4. **(a) 接続が成立した時点の相手の資格情報（その後で相手が資格情報を変えても追随しない）。**
   **(b) PID は使い回されるから。** あるプロセスが終われば同じ PID が別プロセスへ割り当てられうるので、
   長期的な身元には使えない。認可の判断に使うのは主に UID（§6.2）。

5. **`SO_PEERCRED` は認証の材料（相手の資格情報を提供する）である。** それを使って bifrauth が「UID が
   root なら接続を受け付ける」と判断する部分は、資格情報に基づく**接続レベルの認可**にあたる（`serve.rs`
   もこれを認可として扱う）。root 接続に個々の要求・操作を許すかという細かい認可はさらに別の層。また、
   この確認は「相手が root の資格情報を持つ」ことまでで、「`pam_bifrauth` という特定のバイナリだ」ことは
   証明しない。それで十分なのは root 側を TCB に置く前提のためで、root 侵害は防御対象外（§6.2、正本 §4.2）。

6. **「パスの名前」に対して確認すると、確認と実際の使用の間に対象をすり替えられる隙（TOCTOU）が残るから。**
   名前を開き直すたびにすり替えられうる。一度開いた実体（fd）に対して確認し、そのまま使えば、確認と使用の
   間の隙がなくなる（§6.3）。

7. **`O_NOFOLLOW` は symlink を辿らない（別の場所へすり替えられるのを防ぐ）、`O_DIRECTORY` はディレクトリ
   でなければ失敗する（想定と違う種別を掴まされるのを防ぐ）。** 最後に読むファイルで `nlink==1` を確かめる
   のは、想定外のハードリンク（同じ実体を指す別の名前）が張られていないかを見るため（§6.3、正本 §8.3）。

8. **その場で上書きすると、書いている途中でマシンが落ちれば半分だけ書かれた壊れたファイルが残るから。**
   一時ファイルへ書いてから本番の名前へ rename すると、rename は（同一ファイルシステム内でなら。ここでは
   同じディレクトリなので満たされる）名前を古い実体から新しい実体へ不可分に切り替えるので、読む側からは
   切り替え前（古い内容）か後（新しい内容）のどちらかしか見えず、半分書かれた状態は見えない（§6.4）。

9. **rename や新しい名前の作成は、その事実がファイルの中身ではなく「その名前を持つディレクトリ」に記録
   されるため、中身だけ `fsync` してもディレクトリを `fsync` しないと、名前の変更がディスクに残らないまま
   落ちることがあるから。** bifrauth の現実装では、既存の devices ディレクトリ内でのレコード公開について
   は file＋dir の `fsync` が済んでいる（durability 実装済み）が、初回に `/etc/bifrauthd/users/<uid>/devices`
   の階層を新規作成する際の親ディレクトリ群の `fsync` は未実装で、階層作成まで含めた crash consistency は
   まだ実装済みとは言えない（§6.4）。

10. **防げる「巻き戻し」: 正規の管理 API を通じた失効済み端末の復活（tombstone が防ぐ）と、破損を口実にした
    自動初期化（fail closed が防ぐ）。** 防げない「巻き戻し」: レジストリを書ける主体（root やディスクに
    触れるオフライン攻撃者）が、以前保存された本物の古いスナップショットをまるごと戻すこと。これを検出する
    には単調カウンタ・外部アンカー・認証付き世代番号などが要るが現状は持たず、root 書き換えはそもそも防御
    対象外（§6.4、正本 §4.2）。tombstone/fail closed は一般的なロールバック耐性まで保証するものではない。
