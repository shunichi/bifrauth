# 第 1 章 Linux の最小モデル

> **この章の実装状態:** この章は特定の実装に依存しない Linux の一般知識を扱う。
> bifrauth のコードは第 II 部から読む。

bifrauth のレビューでは「root だけが書ける」「非特権プロセスが侵害されても」「接続してきた相手の UID を確かめる」といった言い回しが繰り返し出てくる。
これらはどれも、Linux がユーザーとプロセスとファイルをどう区別しているか、という土台の上に立っている。
その土台を先に置いておかないと、後の章で「なぜそれが安全なのか」を自分で判断できない。

この章では、その最小限のモデルだけを扱う。
ネットワークやシェルの使い方には踏み込まず、あとの議論で必要になる部品（ユーザー、プロセス、ファイルの権限、常駐サービス）に絞る。
初学者向けに簡略化したモデルを先に示すため、いくつかの箇所では「本書ではこう略す」という但し書きを付ける。
略した先に何があるかを知っておくと、後の章で誤解しにくい。

## 1.1 ユーザーと UID

Linux を使う人間には、`alice` のような**ユーザー名**が割り当てられている。
しかしカーネル（OS の中核）が権限の判断に使うのは名前ではなく、**UID**（user identifier）と呼ばれる数値である。
`alice` という名前は人間のための見た目で、内部ではたとえば UID `1000` のように数値へ対応づけられている。

名前と数値の対応は、`/etc/passwd` のようなアカウント情報のデータベースが持っている。
企業環境では LDAP や Active Directory が同じ役割を担うこともある。
どこにデータがあっても、「この名前はどの UID か」を引く共通の仕組みを **NSS**（Name Service Switch）と呼ぶ。
NSS の詳細は後の章（`resolver.rs` を読む第 19 章）で扱うが、ここでは「名前より UID のほうが権限の一次的な根拠である」とだけ押さえておく。

特別な UID が一つある。
**root** は UID `0` を持つ管理者で、通常の権限チェック（後述するファイルのパーミッションによる判定）をほぼすべて素通りできる。
つまり root は、他のユーザーのファイルを読み書きし、任意のプロセスを止め、システム全体の設定を変えられる。
一般ユーザー（UID が `0` でないユーザー）は、自分に許された範囲でしか操作できない。

ただし「root なら文字どおり何でもできる」わけではない。
本書が主に扱う通常のパーミッションモデル（後述する DAC）では root は非常に強い権限を持つが、Linux には capability、LSM（SELinux/AppArmor など）、名前空間、ファイルの immutable 属性といった追加の制約機構もあり、それらが root の操作を制限する場合がある。
本書では「root は本書のモデルの中で強い権限を持つ」とだけ理解しておき、「万能」とは考えないでおく。

この非対称性が、以降のすべての議論の出発点になる。
「root は強い権限を持つ」という前提があるからこそ、bifrauth は「root が持つ鍵は一般ユーザーには読めない」という保証を土台に組み立てられる。

## 1.2 プロセスと権限

**プロセス**とは、実行中のプログラムのことである。
同じプログラムを二回起動すれば、別々のプロセスが二つできる。

プロセスには、権限判断のもとになる UID が結び付いている。
`alice` がエディタを起動すれば、そのプロセスは UID `1000` の権限で動き、root が起動したプロセスは root の権限で動く。
正確には、カーネルがアクセス可否に使うのは **実効 UID**（effective UID）であり、所属するグループ（実効 GID や補助 GID）や capability なども判断に関わる。
本書では以降、簡略化して実効 UID を単に「プロセスの UID」と呼ぶ。
細部を省いても、後の議論の骨格は変わらない。

ここで大事なのは、権限がユーザーそのものではなくプロセス単位で判断される、という点である。
同じ `alice` のログインセッションの中にも、`alice` 権限で動くプロセスと、後述する管理サービス（root 権限）へ要求を出すプロセスが混在しうる。
ユーザーが同じでも、プロセスが持つ権限は一律ではない。

プロセスの権限は、原則として起動元から引き継がれ、プロセスが自分の判断で勝手に上げることはできない。
一般ユーザーのプロセスが独断で root になることはない。
ただし例外として、`setuid` された実行ファイル、`sudo`、systemd によるサービス起動といった、あらかじめ用意された認可済みの仕組みを通れば、起動元と異なる資格情報で動くプロセスが生まれる。
これらは「勝手な昇格」ではなく、管理者が明示的に許した経路である。
権限を越える操作は、こうした正規の経路を通って、特権側のプロセスへ依頼する形を取る。
この「依頼する側（非特権）」と「判断する側（特権）」の分離こそ、bifrauth が守ろうとしている構造である。

プロセスがファイルやネットワーク接続を開くと、カーネルはそれを **ファイルディスクリプタ**（file descriptor）という小さな整数で参照させる。
開いたファイルは `3`、開いたソケットは `4`、といった具合に番号が割り振られ、プロセスはその番号を通じて読み書きする。
ファイルディスクリプタはプロセスごとの番号だが、`fork`/`exec` による子プロセスへの継承や、systemd からサービスへの受け渡しによって、別のプロセスへ引き継がせることもできる。
この「開いた資源を別プロセスへ渡せる」性質は、後で systemd のソケットアクティベーション（§1.4）を理解する伏線になる。

## 1.3 ファイルの所有者とパーミッション

Linux は多くのものをファイルとして扱う。
設定も、鍵の保存先も、プロセス間の通信口（ソケット）も、ファイルシステム上の名前を持つ。
したがって「誰がそのファイルを読めるか、書けるか」を決める仕組みが、そのままシステムの安全性を左右する。

各ファイルには、**所有者**（owner）となるユーザーと、**グループ**（group）が結び付いている。
そのうえで、アクセス権を三つの区分で持つ。

- **owner**：その所有者本人に対する許可
- **group**：所有グループのメンバーに対する許可
- **other**：所有者でもなく所有グループのメンバーでもない全員に対する許可

区分ごとに、読み取り（read）、書き込み（write）、実行（execute）の三種類の許可を独立に設定できる。
この設定をまとめて **パーミッション**（permission）または **mode** と呼ぶ。
この「所有者と mode だけでアクセスを決める」方式を **DAC**（discretionary access control、任意アクセス制御）と呼ぶ。
`0640` のような数字で表記されることが多く、これは「所有者は読み書き、グループは読み取りのみ、その他は一切不可」を意味する。

通常ファイルでは read/write/execute の意味は直感どおりだが、**ディレクトリ**では意味が変わるので注意する。
ディレクトリに対する read は「中の名前の一覧を得る」、write は「エントリの作成や削除（＝中のファイルを追加・削除・改名する）」、execute は「そのディレクトリを通り抜けて中の名前へ到達する（traverse）」を指す。
中のファイルを作ったり消したりするには、通常 write と execute の両方が要る。
この区別は、bifrauth が鍵置き場を `root:root 0700` のディレクトリ階層で守る理由（第 16 章）に直結する。

パーミッションのうち、レビューで最も注意すべきは **write** である。
あるファイルを書き換えられる者は、そのファイルの内容を改ざんできる。
たとえば認証に使う公開鍵の一覧を一般ユーザーが書き換えられるなら、攻撃者は自分の鍵を勝手に登録できてしまう。

ただし、ファイル自身の mode だけを見て安心してはいけない。
あるファイルが `root:root 0640` であっても、そのファイルを含む**親ディレクトリ**を攻撃者が書けるなら、攻撃者は元のファイルを削除（unlink）したり改名（rename）したりして、自分が用意した別のファイルへ置き換えられる。
つまり守るべきは一つのファイルではなく、そのファイルへ至るパスを構成するディレクトリすべてである。
さらに、途中に細工されたシンボリックリンクが挟まっていないか、開く直前と開いた後で対象がすり替わっていないか（TOCTOU、time-of-check to time-of-use）にも注意が要る。
正本設計書 §8.2 が、パスを構成する各ディレクトリの種別・所有者・mode の検証や、`O_NOFOLLOW` による symlink 回避、`fstat` による確認を要求しているのはこのためである。

プロセス間通信に使う Unix ソケットも、この観点では「内容を書き換えるファイル」とは少し違う扱いになる。
ソケットそのもの（inode）の mode は「誰がそこへ接続してよいか」に関わり、ソケットを置いた親ディレクトリの mode は「その名前を誰がすり替えられるか」に関わる。
両者は別の保護であり、後の章では両方を確認する。

## 1.4 サービスとデーモン、そして systemd

これまで見たプロセスは、ユーザーが起動して使い終われば消える。
一方で、要求が来るのを待ち続けて常駐するプロセスもある。
こうした常駐プロセスを **デーモン**（daemon）と呼び、提供する機能の単位を **サービス**（service）と呼ぶ。

デーモンをいつ起動し、落ちたら再起動し、どの権限で動かすかを管理するのが **systemd** である。
systemd は Linux の起動直後から動き、各サービスを定義に従って立ち上げる。
bifrauth の判断を担う中核デーモンも、root 権限のサービスとして systemd に管理される。

systemd には、サービス本体を常時起動しておかず、通信口（ソケット）だけを先に用意しておく仕組みがある。
これを **ソケットアクティベーション**（socket activation）と呼ぶ。
bifrauth が使う方式では、systemd が先に接続待ち受け用のソケットを作って listen しておき、最初の接続が来た時点でサービス本体を起動して、その**待ち受けソケットそのもの**をファイルディスクリプタとしてサービスへ渡す。
接続を受け付ける（accept する）のはサービス本体であり、サービスが起動するまでの接続はカーネルの待ち行列（backlog）で待たされる。
（別方式として、接続ごとに systemd 側が accept 済みの接続をサービスへ渡す設定もあるが、bifrauth はそれを使わない。）
この仕組みは後の章（第 19 章）で bifrauth のコードに沿って詳しく見る。
ここでは「常駐サービスと、その起動を管理する systemd があり、待ち受けソケットをサービスへ渡せる」という地図だけ持っておけばよい。

## 1.5 なぜ「root だけが書ける」が安全性の土台になるのか

ここまでの部品を一つにつなぐと、bifrauth の設計を読むための最初の補助線が引ける。

bifrauth の脅威モデルは、「一般ユーザーのセッションは侵害されうる」と想定する。
つまり、`alice` の権限で動くプロセスの中に、攻撃者が仕込んだ悪意あるプログラムが混ざっているかもしれない、と考える。
一方で、root 側（root 権限で動くコードと、root だけが書けるデータ）は、侵害されていない前提で信頼する土台、いわゆる **TCB**（trusted computing base、信頼される計算基盤）として扱う。

ここで、正本設計書との言葉の対応に注意する。
正本 §3 は「root 権限を持つ攻撃者は信頼しない」と書き、§4.2 は「root 権限を取得した攻撃者」を防御対象外に挙げている。
これは「root 側を信頼しない」という意味ではなく、「root がすでに乗っ取られている状況は、この設計が守る範囲の外だ」という意味である。
言い換えると、root が侵害されていない限り、bifrauth は非特権側の侵害から認証を守る。
root まで奪われたら、それはもはや bifrauth の防御範囲ではない。

この前提から、防御の方針が決まる。
認証の可否を最終的に判断するコードと、その判断が依存するデータ（登録済み鍵など）を、すべて root 側（root 所有かつ一般ユーザー書き込み不可のパス）へ置く。
そうすれば、一般ユーザー権限のマルウェアは、判断ロジックもデータも書き換えられない。
マルウェアにできるのは、せいぜい root 側へ要求を出すことだけであり、要求の中身は root 側で検証される。

ただし §1.3 で見たとおり、「ファイル一つを `root:root 0640` にする」だけでは壁にならない。
そのファイルへ至る親ディレクトリまで非特権者が書けないこと、そして開くときに symlink のすり替えや TOCTOU を避ける安全な手順を踏むこと、この両方がそろって初めて改ざんを防げる。
壁になるのは一行のパーミッションではなく、「ファイルと親ディレクトリを含む保護されたパス」と「安全にそこを開く手続き」の組である。
この壁がどこに引かれ、どこまで正しく守られているかを追う作業が、bifrauth のセキュリティレビューの中心の一つになる。
次章では、この「侵害されうる側」と「信頼する側」を分ける考え方そのもの、すなわち認証と認可、そして脅威モデルの立て方を扱う。

## 理解度クイズ

> 解答と解説は付録 B にまとめてある。まず自分で考えてから確認してほしい。

1. カーネルがユーザーの権限を判断するとき、一次的な根拠にするのはユーザー名か、それとも別のものか。別のものなら、それは何か。
2. root（UID `0`）は本書のパーミッションモデルの中で一般ユーザーとどう違うか。また「root なら何でもできる」と言い切れない理由を一つ挙げよ。
3. 「権限はユーザー単位ではなくプロセス単位で判断される」とはどういう意味か。同じユーザーのログインセッションを例に説明せよ。また、プロセスが起動元と異なる資格情報で動くのはどのような場合か。
4. あるファイルのパーミッションが `0640`、所有者が root、所有グループが root で、対象の一般ユーザーは root グループに属さず、追加の ACL も設定されていないとする。このとき、その一般ユーザー（root 自身や特権を迂回する capability は考えない）はそのファイルに対して何ができて、何ができないか。
5. bifrauth は「一般ユーザーのセッションは侵害されうるが、root 側は侵害されていない前提で信頼する（TCB として扱う）」と仮定する。この仮定から、認証の最終判断とその依存データをどちら側に置くべきという方針が導かれるか。理由も述べよ。
6. あるデータファイルを `root:root 0640` にしたうえで、その内容が非特権ユーザーに改ざんされないことを保証したい。ファイル自身の mode を正しく設定するだけでは不十分な理由を、親ディレクトリに触れて説明せよ。
7. systemd のソケットアクティベーションで、bifrauth が使う方式では、systemd はサービス本体へ何を渡すか。「接続そのもの」と「待ち受けソケット」のどちらか、また接続を accept するのはどちらか（systemd か、サービス本体か）を答えよ。
