# 第 1 章 Linux の最小モデル

> **この章の実装状態:** この章は特定の実装に依存しない Linux の一般知識を扱う。
> bifrauth のコードは第 II 部から読む。

bifrauth のレビューを読み始めると、同じ言い回しに何度もぶつかる。
「root だけが書ける」。「非特権プロセスが侵害されても」。「接続してきた相手の UID を確かめる」。
どれも、これさえ押さえれば安全だ、と言っているように見える。
本当にそうか。
それを自分で判断するには、言い回しの下にある Linux の区別の仕方を知っていないといけない。

区別の対象は三つしかない。
ユーザーと、プロセスと、ファイルである。
この章では、その三つと、常駐サービスを動かす仕組みだけを扱う。
シェルの使い方やネットワークには立ち入らない。
初学者向けに簡略化する箇所には「本書ではこう略す」と断りを入れる。略した先に何があるかを一言添えておくと、後の章で足をすくわれない。

## 1.1 ユーザーと UID

Linux を使う人間には、`alice` のような**ユーザー名**が割り当てられている。
だがカーネル（OS の中核）が権限判断に使うのは、名前ではない。
使うのは **UID**（user identifier）という数値である。
`alice` は人間のための見た目にすぎず、内部ではたとえば UID `1000` のように数値へ対応づけられている。

名前と数値の対応は、`/etc/passwd` のようなアカウント情報のデータベースが持っている。
企業環境では LDAP や Active Directory が同じ役割を担うこともある。
どこにデータがあっても、「この名前はどの UID か」を引く共通の仕組みを **NSS**（Name Service Switch）と呼ぶ。
NSS は後の章（`resolver.rs` を読む第 19 章）で扱う。
ここでは、権限の一次的な根拠は名前ではなく UID のほうだ、とだけ押さえておけばよい。

特別な UID が一つある。
**root** は UID `0` を持つ管理者で、通常のパーミッション判定（このあと §1.3 で見る）をほぼ素通りできる。
他人のファイルを読み書きし、任意のプロセスを止め、システム全体の設定を変えられる。
一般ユーザー、つまり UID が `0` でないユーザーは、自分に許された範囲でしか動けない。

では root は文字どおり何でもできるのか。
そうではない。
本書が主に扱う通常のパーミッションモデル（後述する DAC）の中では root は確かに非常に強い。
だが Linux には capability、LSM（SELinux や AppArmor）、名前空間、ファイルの immutable 属性といった追加の制約もあり、それらが root の操作を止める場合がある。
本書では「root は本書のモデルの中で強い」とだけ理解し、「万能」とは考えないでおく。

この非対称が、以降のすべての出発点になる。
root が強い権限を持つからこそ、bifrauth は「root が持つ鍵は一般ユーザーには読めない」という一点を土台にできる。

## 1.2 プロセスと権限

**プロセス**（process）とは、実行中のプログラムのことである。
同じプログラムを二回起動すれば、別々のプロセスが二つできる。

プロセスには、権限判断のもとになる UID が結び付いている。
`alice` がエディタを起動すれば、そのプロセスは UID `1000` で動く。
正確には、カーネルがアクセス可否に使うのは **実効 UID**（effective UID）で、所属グループや capability なども判断に関わる。
本書では以降、簡略化して実効 UID を単に「プロセスの UID」と呼ぶ。
細部を省いても、これから組み立てる骨格は変わらない。

ここで肝心なのは、カーネルが権限を判断する単位である。
判断されるのはユーザーではなく、その操作を実際に行ったプロセスの**資格情報**（credentials、UID や GID など権限判断のもとになる素性）である。
この一点を取り違えると、後の PAM や権限委譲の話がすべてずれてしまう。

具体的に見てみよう。
`alice` の使うシェルやエディタは、`alice` の資格情報で動く。
そのシェルから、root 権限で動くサービスへ何かを依頼したとする。
このとき、依頼した側のプロセスの権限は上がるだろうか。
上がらない。
依頼のために**プロセス間通信**（IPC、inter-process communication）をしただけであり、依頼した `alice` のプロセスは `alice` のままである。
root のサービスは、それとは別のプロセスとして、自分の資格情報で要求を検証し実行する。
つまり、要求を投げる側と、それを判断する側は、はじめから別々のプロセスであり、別々の権限を持つ。

では、同じ `alice` のログイン操作から、`alice` と違う資格情報で動くプロセスが生まれることはあるのか。
ある。
ただし、勝手には起きない。
`setuid` された実行ファイル、`sudo`、systemd によるサービス起動といった、あらかじめ管理者が許した経路を通ったときだけである。
通常のシェルは `alice`（実効 UID `1000`）で動き、`sudo` 経由で生成された別プロセスは実効 UID `0` で動く。
同じログインの中でも、経路が違えば資格情報が違う。
これは勝手な昇格ではなく、管理者が明示的に許した権限移行（昇格）の経路である。

この「依頼する側（非特権）」と「判断する側（特権）」の分離こそ、bifrauth が守ろうとしている構造そのものである。

最後に、プロセスが資源をどう掴むかにも触れておく。
プロセスがファイルやネットワーク接続を開くと、カーネルはそれを **ファイルディスクリプタ**（file descriptor）という小さな整数で参照させる。
開いたファイルは `3`、開いたソケットは `4`、といった具合に番号が割り振られ、プロセスはその番号を通じて読み書きする。
番号はプロセスごとのものだが、`fork`/`exec` による子プロセスへの継承や、systemd からサービスへの受け渡しで、別のプロセスへ引き継がせられる。
この「開いた資源を別プロセスへ渡せる」性質は、§1.4 のソケットアクティベーションで効いてくる。

## 1.3 ファイルの所有者とパーミッション

Linux は多くのものをファイルとして扱う。
設定も、鍵の保存先も、プロセス間の通信口（ソケット）も、ファイルシステム上の名前を持つ。
だから「誰がそれを読めるか、書けるか」を決める仕組みが、そのまま安全性を左右する。

各ファイルには、**所有者**（owner）となるユーザーと、**グループ**（group）が結び付いている。
そのうえで、アクセス権を三つの区分で持つ。

- **owner**：その所有者本人に対する許可
- **group**：所有グループのメンバーに対する許可
- **other**：所有者でも所有グループのメンバーでもない全員に対する許可

区分ごとに、読み取り（read）、書き込み（write）、実行（execute）を独立に設定できる。
この設定をまとめて **パーミッション**（permission）または **mode** と呼ぶ。
所有者と mode だけでアクセスを決めるこの方式を **DAC**（discretionary access control、任意アクセス制御）と呼ぶ。
`0640` のような数字で表され、これは「所有者は読み書き、グループは読み取り、その他は不可」を意味する。

通常ファイルなら read/write/execute の意味は直感どおりだが、**ディレクトリ**では意味が変わる。
ここは取り違えやすい。
ディレクトリに対する read は「中の名前の一覧を得る」、write は「エントリの作成や削除、つまり中のファイルを追加・削除・改名する」、execute は「そのディレクトリを通り抜けて中の名前へ到達する（traverse）」を指す。
中のファイルを作ったり消したりするには、ふつう write と execute の両方が要る。
この区別が、bifrauth が鍵置き場を `root:root 0700` のディレクトリ階層で守る理由（第 18 章）に直結する。

パーミッションのうち、レビューでまず目を向けるべきは **write** である。
あるファイルを書き換えられる者は、その内容を改ざんできる。
認証に使う公開鍵の一覧を一般ユーザーが書き換えられるなら、攻撃者は自分の鍵を勝手に登録できてしまう。
ならば、そのファイルを `root:root 0640` にして一般ユーザーの write を落とせば、それで改ざんは防げる。

そう考えたくなる。
だが、それでは足りない。
ファイル自身が `root:root 0640` でも、そのファイルを含む**親ディレクトリ**を攻撃者が書けるなら、話は変わる。
攻撃者は元のファイルを削除（unlink）したり改名（rename）したりして、自分の用意した別ファイルへ丸ごとすり替えられる。
中身を書き換えられなくても、名前ごと差し替えれば同じことだ。
つまり守るべきは一つのファイルではなく、そのファイルへ至るパスを構成するディレクトリすべてである。
さらに、途中に細工された**シンボリックリンク**（symlink、別の場所を指す名前）が挟まっていないか、開く直前と開いた後で対象がすり替わっていないか（TOCTOU、time-of-check to time-of-use）にも気を配る必要がある。
正本設計書 §8.2 が、パスを構成する各ディレクトリの種別・所有者・mode の検証や、`O_NOFOLLOW` による symlink 回避、`fstat` による確認を要求しているのは、この抜け道をふさぐためである。

**Unix ドメインソケット**（Unix domain socket、同じマシン内のプロセスをつなぐ通信路。第 6 章）は、この観点では「内容を書き換えるファイル」とは少し違う扱いになる。
ソケットそのもの（inode）の mode は「誰がそこへ接続してよいか」に関わり、それを置いた親ディレクトリの mode は「その名前を誰がすり替えられるか」に関わる。
別々の保護であり、後の章では両方を確かめる。

## 1.4 サービスとデーモン、そして systemd

これまでのプロセスは、ユーザーが起動して使い終われば消えた。
だが、要求が来るのを待ち続けて常駐するプロセスもある。
こうした常駐プロセスを **デーモン**（daemon）と呼び、提供する機能の単位を **サービス**（service）と呼ぶ。

デーモンをいつ起動し、落ちたら再起動し、どの権限で動かすか。
これを一手に管理するのが **systemd** である。
systemd は Linux の起動直後から動き、各サービスを定義どおりに立ち上げる。
bifrauth の判断を担う中核デーモンも、root 権限のサービスとして systemd に管理される。

systemd には、サービス本体を常時起動しておかず、通信口だけを先に用意する仕組みがある。
これを **ソケットアクティベーション**（socket activation）と呼ぶ。
bifrauth が使う方式では、systemd が先に接続待ち受け用のソケットを作って listen しておく。
最初の接続が来た時点でサービス本体を起動し、その**待ち受けソケットそのもの**をファイルディスクリプタとして本体へ渡す。
接続を受け付ける（accept する）のはサービス本体であり、本体が起動するまでの接続はカーネルの待ち行列（backlog）で待たされる。
（接続ごとに systemd 側が accept 済みの接続を渡す別方式もあるが、bifrauth はそれを使わない。）
この仕組みは第 19 章で bifrauth のコードに沿って見る。
ここでは、待ち受けソケットをサービスへ渡せる、という一点だけ持っておけばよい。

## 1.5 なぜ「root だけが書ける」が安全性の土台になるのか

ここまでの部品をつなぐと、bifrauth の設計を読むための最初の補助線が引ける。

bifrauth の脅威モデルは、一般ユーザーのセッションは侵害されうる、と想定する。
`alice` の権限で動くプロセスの中に、攻撃者が仕込んだプログラムが混ざっているかもしれない。
一方で root 側、すなわち root 権限で動くコードと root だけが書けるデータは、侵害されていない前提で信頼する。
この信頼する土台を **TCB**（trusted computing base、信頼される計算基盤）と呼ぶ。

ここで、正本設計書との言葉の食い違いに気をつけたい。
正本 §3 は「root 権限を持つ攻撃者は信頼しない」と書き、§4.2 は「root 権限を取得した攻撃者」を防御対象外に挙げている。
一見すると、root を信頼するという先ほどの説明と逆に読める。
だが矛盾はない。
これは「root 側を信頼しない」という意味ではなく、「root がすでに乗っ取られた状況は、この設計が守る範囲の外だ」という意味である。
root が侵害されていない限り、bifrauth は非特権側の侵害から認証を守る。
root まで奪われたら、それはもう bifrauth の防御範囲ではない。

この前提から、方針が決まる。
認証の可否を最終的に判断するコードと、その判断が依存するデータ（登録済み鍵など）を、すべて root 側へ置く。
そうすれば、一般ユーザー権限のマルウェアは、判断ロジックもデータも書き換えられない。
できるのは root 側へ要求を出すことだけで、その中身は §1.2 で見たとおり、別プロセスである root 側が自分の資格情報で検証する。

もっとも、§1.3 で一度崩したとおり、「ファイルを `root:root 0640` にする」だけでは壁にならない。
そのファイルへ至る親ディレクトリまで非特権者が書けないこと。
そして開くときに symlink のすり替えや TOCTOU を避ける安全な手順を踏むこと。
この両方がそろって、はじめて改ざんを防げる。
壁になるのは一行のパーミッションではなく、「ファイルと親ディレクトリを含む保護されたパス」と「安全にそこを開く手続き」の組である。

この壁がどこに引かれ、どこまで正しく守られているか。
それを一つずつ追う作業が、bifrauth のセキュリティレビューの中心になる。
次の章では、いま「侵害されうる側」と「信頼する側」と呼び分けたその線を、正面から扱う。
認証と認可の違い、そして脅威モデルの立て方である。

## 理解度クイズ

> 解答と解説は付録 B にまとめてある。まず自分で考えてから確認してほしい。

1. カーネルがユーザーの権限を判断するとき、一次的な根拠にするのはユーザー名か、それとも別のものか。別のものなら、それは何か。
2. root（UID `0`）は本書のパーミッションモデルの中で一般ユーザーとどう違うか。また「root なら何でもできる」と言い切れない理由を一つ挙げよ。
3. 「権限は、操作を行ったプロセスの資格情報で判断される」とする。(a) `alice` のプロセスが root のサービスへ IPC で依頼したとき、依頼した側のプロセスの権限は上がるか。理由も述べよ。(b) 同じ `alice` のログイン操作を起点に、`alice` と異なる資格情報（実効 UID `0` など）で動くプロセスが生まれるのは、どういう場合か。
4. あるファイルのパーミッションが `0640`、所有者が root、所有グループが root で、対象の一般ユーザーは root グループに属さず、追加の ACL も設定されていないとする。このとき、その一般ユーザー（root 自身や特権を迂回する capability は考えない）はそのファイルに対して何ができて、何ができないか。
5. bifrauth は「一般ユーザーのセッションは侵害されうるが、root 側は侵害されていない前提で信頼する（TCB として扱う）」と仮定する。この仮定から、認証の最終判断とその依存データをどちら側に置くべきという方針が導かれるか。理由も述べよ。
6. あるデータファイルを `root:root 0640` にしたうえで、その内容が非特権ユーザーに改ざんされないことを保証したい。ファイル自身の mode を正しく設定するだけでは不十分な理由を、親ディレクトリに触れて説明せよ。
7. systemd のソケットアクティベーションで、bifrauth が使う方式では、systemd はサービス本体へ何を渡すか。「接続そのもの」と「待ち受けソケット」のどちらか、また接続を accept するのはどちらか（systemd か、サービス本体か）を答えよ。
