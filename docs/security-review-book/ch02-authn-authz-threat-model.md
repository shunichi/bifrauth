# 第 2 章 認証と認可、脅威モデルの立て方

> **この章の実装状態:** 一般知識を扱う章だが、bifrauth 固有の記述には節ごとにラベルを付ける。
> bifrauth 固有部分は〔仕様（正本）。iOS アプリ・Secure Enclave・transport など**未実装領域を含む**。
> 参照時点: 設計書 v0.3.3 / commit `e8f8c6e`（2026-07-18）〕であり、実証済みの保証ではなく設計上の
> 意図として読む。

前章で、信頼する側と信頼しない側という線を引いた。
root 側は信頼し、一般ユーザーのセッションは侵害されうるものとして扱う。
だが、その線を引いただけでは、まだ何も守れていない。
線の内側で「この人は本人だ」とどうやって確かめるのか。
確かめたうえで「この操作を許してよいか」をどう決めるのか。

この二つは、よく一語で「認証」と呼ばれてしまう。
実際には別の問いであり、bifrauth のレビューでは常に分けて考える必要がある。
この章では、まずその二つを分ける。
次に、パスワード認証が実際に何をしているかを覗き、そこに潜むなりすましとリプレイという穴を見る。
最後に、何を誰から守るのかを決める脅威モデルの立て方を扱う。

---

## 2.1 認証と認可 — 何を確かめ、何を許すか

**認証**（authentication）は、「あなたは誰か」を確かめることである。
**認可**（authorization）は、「ある操作を許すか」を決めることである。
この二つはよく「認証してから認可する」という順で語られる。
典型的にはそのとおりで、誰かを確かめた結果が、許すかどうかの判断材料になる。

ただし、常に認証が先とは限らない。
認可は、相手が誰か分からないまま下されることもある。
たとえば「誰でも入ってよいロビー」は、本人確認なしに立ち入りを許す認可の一種である。
逆に、機密区画への立ち入りを判断するときだけ本人確認を求める、という運用もある。
つまり認可は、必要に応じて認証の結果を入力に使うのであって、いつも認証を待つわけではない。

bifrauth の登場人物に当てはめると、役割がはっきり分かれる。
Face ID を起点とする本人確認は、認証の一部を担う。
ここで場所を混同しないでおきたい。
bifrauth のモジュールは Linux 側で PAM から呼ばれるが、Face ID の判定と Secure Enclave の署名そのものは iPhone 上で行われる（第 3 章）。
一方で「この polkit の操作を実行してよいか」を決める認可は、polkit が担う（第 4 章）。

ここで、認証が何を確かめているのかを正確にしておきたい。
「Face ID が `alice` を見分ける」と考えると、後の章でつまずく。
〔仕様（正本）〕bifrauth の認証根拠は、Face ID そのものではない。
`alice` の uid に登録された端末の鍵が、root 側の verifier が作ったその場かぎりの要求（誰・どのサービス・どの端末・どの期限に向けたものかが刻まれている）へ**署名**（signature）し、その署名が検証を通ることである。
Face ID は、その署名に使う Secure Enclave の鍵の使用を、その場で許可するゲートにすぎない。
本人性を最終的に支えるのは、顔の一致という主張ではなく、登録済みの鍵でしか作れない署名が検証を通った、という事実である。

もう一つ、混同を避けたい区別がある。
polkit の文脈には、「その操作を要求してきたプロセスの主体」と、「PAM が本人確認する対象の identity」という二つの主体が登場する。
この二つは必ずしも同じではなく、後の章では取り違えないよう分けて扱う。

## 2.2 パスワード認証は実際に何をしているか

もっとも身近な認証は、パスワードである。
仕組みは単純に見える。
ユーザーが名前とパスワードを入力し、システムがそれを照合し、合っていれば本人と認める。

だが「照合する」の中身を覗くと、注意点が見えてくる。
まともに設計されたシステムは、パスワードそのものを保存しない。
かといって、汎用の高速なハッシュを一回かけるだけでも足りない。
高速なハッシュは、辞書攻撃や総当たりで大量に試すのが容易だからだ。
そこで、ユーザーごとに異なる **ソルト**（salt、同じパスワードでも保存値が変わるようにする追加データ）と、計算を意図的に重くする **コスト**（cost）を組み込んだ、パスワード専用の変換（パスワードハッシュや KDF、Linux なら shadow の方式）を使う。
ログイン時には、入力されたパスワードから同じ変換をやり直し、保存された値と一致するかを見る。

それでも、保存値が漏れれば、攻撃者は手元で候補を延々と試せる（オフライン推測）。
ソルトとコストはこの推測を遅くするが、弱いパスワードなら破られうる。
だから「保存値からは復元しにくい」だけで安全とは言い切れない。

問題は、保存のしかたよりも、パスワードが「通る」瞬間に集まる。
典型的な方式では、入力の瞬間にどこかで平文のパスワードが扱われる。
偽のログイン画面に入力させられたり、端末そのものが乗っ取られていたり、通信が暗号化されていなかったりすれば、その平文は盗まれ、そして次も同じように通ってしまう。
（暗号化された通信なら途中の第三者には見えないし、PAKE のように秘密そのものをサーバへ渡さない方式もある。ここで問題にしているのは、平文が扱われる端点や不安全な経路である。）
ここに、認証の二つの基本的な穴が口を開ける。

## 2.3 なりすましとリプレイ

**なりすまし**（impersonation）は、攻撃者が正規のユーザーとして扱われてしまう、という結果を指す。
盗んだパスワードを入力すれば、システムから見れば本人と区別がつかない。
どうやってそこへ至るかは一つではないが、結果として本人の座に座られることを、こう呼ぶ。

**リプレイ**（replay）は、そこへ至る手法の一つである。
過去に成立した正しいやり取りを記録しておき、それをそのまま再送する。
中身を理解する必要はない。
一度通ったものをもう一度流して通るなら、それで足りる。

この二つは、いつも一緒に起きるわけではないし、原因も一つではない。
検証する側が要求について確かめるべき性質は、大きく三つある。
そして、どれが欠けるかで、通ってしまう攻撃の種類が変わる。

- **新しさ**（freshness）：その要求が今回のために新しく作られたか。一回きりの nonce、期限、使用済みの記録（consume）などで確かめる。これが欠けると、同じ文脈の過去の要求をそのまま再送できる（同一文脈のリプレイ）。
- **文脈への束縛**（context / audience binding）：その要求が、この相手・このサービス・この端末・この期限に向けたものか。これが欠けると、ある場面では正しい要求を、別のサービスやユーザーや端末へ流用できる（別文脈への流用）。
- **正しい相手が作った裏付け**（authenticity）：署名などで、要求が本物の発行者のものだと確かめる。これが欠けると、主に偽造や改ざんを許す。

三つはどれも必要で、互いを肩代わりしない。
署名で authenticity を確かめていても、新しさを見ていなければ、正しい署名つきの古い要求がそのまま再送されてしまう。
逆に、署名がなくても、十分な新しさの検査だけで単純な過去の要求の再送は弾ける。
なりすましは、こうした検証の隙や、使い回せる資格情報の露出から生じる結果である。

逆に言えば、対策の方向も決まる。
長期の秘密を手元に持ったまま、その場かぎりの問いかけに鍵で答える方式なら、秘密そのものを送らずに済み、古い答えの再送も弾ける。
この仕掛け（チャレンジレスポンス（challenge-response）と署名）は第 5 章で扱う。

ここで、認証の結果だけを送る方式の危うさを、二つに分けて見ておく。
「この人は認証に成功した」という真偽値を、ネットワーク越しに `true` として送るとしよう。
一つめの危険は、その `true` を盗んで再送すれば通ってしまうこと（リプレイ）。
二つめは、そもそも盗む必要すらなく、攻撃者が自分で `true` を作って送れてしまうこと（偽造、forge）。
危ういのは値がブール型だからではない。
危ういのは、成功したという主張だけを、鍵による証明も、その場かぎりの値も、宛先や用途への結び付きもないまま信じてしまうことである。
（署名を付け、一回きりにし、宛先に束縛したトークンなら設計できる。それは主張を裏付けている点で裸の `true` とは違う。）
〔仕様（正本）§5.1〕bifrauth はこの真偽値を信頼せず、Secure Enclave の鍵による、その場かぎりの要求への署名を検証する。

## 2.4 脅威モデルの立て方

穴が見えてくると、次の問いが自然に湧く。
どこまで守れば「安全」と言えるのか。
すべての攻撃を防ぐ、と言えれば楽だが、それは達成できない約束である。
守る範囲を決めずに「安全」と言うのは、何も言っていないに等しい。

そこで、守る範囲を先に言葉にする。
この作業を **脅威モデル**（threat model）を立てる、と呼ぶ。
立てるときの部品を、次のように置く。

- **資産**（asset）：何を守るのか。bifrauth なら、認証判断の完全性、その根拠となる秘密鍵や登録情報、そして結果として守られる 1Password へのアクセスである。
- **守るべき性質**（security goal / invariant）：資産について何が成り立ってほしいか。たとえば「非特権側が単独で認証成功を作り出せない」。
- **敵**（adversary）：誰が相手で、何ができると想定するか。同じ LAN 上で通信を盗み見る者、一般ユーザー権限を得たマルウェア、といった具体で置く。
- **信頼境界**（trust boundary）：信頼する側と信頼しない側を分ける線。前章で引いた root 側と非特権側の線が、これにあたる。
- **前提と範囲外**（assumptions / out-of-scope）：成り立つと仮定すること、そして初めから守らないと決めることを、明示的に並べる。

最後の一つは軽く見られがちだが、レビューでは特に重要になる。
守らないと決める理由には、いくつか種類があり、混ぜてはいけない。

- **製品としての非ゴール**（non-goal）：そもそも作らない機能（たとえば初期版で遠隔認証を提供しないこと）。
- **前提**（assumption）：成り立つと仮定して設計する条件。たとえば Secure Enclave の鍵は取り出せない、という仮定は正本 §3 の前提である。
- **範囲外の脅威**（out-of-scope threat）：守り得るかどうかにかかわらず、この設計では守らないと宣言した攻撃。正本 §4.2 は、root をすでに奪った攻撃者に加えて、ユーザーが正当な要求を誤って承認してしまうこと、そしてサービス拒否（DoS）攻撃も、明示的に防御対象外に挙げている。
- **残余リスク**（residual risk）：範囲内の対策を施したうえで、なお残る確率や影響。たとえば、要求の取り違えを number matching（第 20 章）で緩和しても、ユーザーが確認を怠れば取り違えは完全には消えない。緩和したうえで受け入れる部分がこれにあたる。

範囲外の脅威と残余リスクは、混ぜやすいので気をつける。
「誤承認」や「DoS」は、正本が最初から守らないと宣言した範囲外の脅威であって、対策後に残る残余リスクではない。
なお正本は DoS を範囲外に置きつつ、レート制限などを多層防御として実装し得るが、それで分類が自動的に残余リスクへ変わるわけではない。
分類は「何を守ると宣言したか」で決まり、実装の有無では決まらない。

範囲外を宣言するのは、能力が足りないからではない。
それは、この設計が何を約束し、何を約束しないかを決める境界そのものである。
たとえば「root を奪った攻撃者は防御対象外」は、次のように具体化できる。
bifrauth では、判断を下す verifier、PAM モジュール、登録情報が信頼境界の内側（TCB、信頼される計算基盤 trusted computing base。第 1 章）にある。
root を奪った攻撃者はそれらを書き換えられるので、この設計が与える「Linux 側の認証成功判定」の保証は、そもそも成り立たなくなる。
守れないものを「守る」と書けば約束が嘘になるから、境界の外に置く。

レビューをするとき、これらは最初に確かめる。
守ると言っている資産と性質は何か。
想定している敵はどこまでか。
線はどこに引かれ、前提と範囲外は明示されているか。
そして、その境界の外にあるものを、うっかり守るふりをしていないか。

## 2.5 非特権プロセスが侵害されても、単独では偽造できない

ここまでの言葉がそろうと、bifrauth の中心的な目標を正確に言い直せる。

前章では、判断とデータを root 側に置く、という方針を導いた。
本章の言葉で言えば、これは認証の判断を信頼境界の内側（TCB）に閉じ込める、ということである。
そのうえで、非特権側が握るものを、なりすましやリプレイの材料にならないよう選ぶ。
使い回せる秘密も、成功を表す真偽値も、非特権側には渡さない。

すると、次の目標を掲げられる。
〔仕様（正本）。以下は設計上の目標であり、実装と運用がすべての不変条件を満たす限りで成り立つ〕
一般ユーザーのセッションが丸ごと侵害され、そこで動くプロセスが攻撃者の思うままになっても、そのプロセスは単独では認証成功を偽造できない。
偽造に必要な鍵は Secure Enclave の中にあり、取り出せないと仮定する（正本 §3。iPhone OS や Secure Enclave の完全な侵害は §4.2 で範囲外）。
判断は root 側が下し、送られる証はその場かぎりで使い回せない。

ただし、この目標を「攻撃者にはもう何もできない」と読み替えてはいけない。
非特権側の攻撃者にも、まだできることは残っている。
これらを、前節の分類に沿って区別しておく。

- 正規の認証要求を何度も誘発すること。これ自体は攻撃の足がかりで、次の取り違えや承認疲れにつながる。
- Face ID の手軽さにつけ込み、ユーザーの承認疲れや要求の取り違えを誘うこと。「誤承認」は正本 §4.2 の範囲外の脅威であり、number matching で緩和しても消えない部分は残余リスクとして残る。
- 通信を妨げてサービスを止めること（DoS）。これは正本 §4.2 の範囲外の脅威である。

そして root まで奪われれば、そもそも保証は成り立たない（これも §4.2 の範囲外）。
「単独では偽造できない」は、これらの条件と限界の上に立つ目標であって、無条件の安全宣言ではない。

この目標を、どの部品がどう支え、どこに未実装や運用上の前提が残っているか。
それを一つずつ確かめるのが、この本の後半でやることである。
次の章では、その認証を Linux 側で実際に呼び出す枠組み、PAM を扱う。
「まず本人確認し、失敗したらパスワードへ戻す」という流れが、どこで組み立てられているのかを見る。

## 理解度クイズ

> 解答と解説は付録 B にまとめてある。まず自分で考えてから確認してほしい。

1. 認証（authentication）と認可（authorization）は何が違うか。また「常に認証が先に行われる」と言えない理由を、認可の例を挙げて説明せよ。
2. bifrauth の認証根拠は「Face ID が `alice` を見分けること」だと言うと、なぜ不正確か。正しくは何が本人性を支えているかを述べよ。
3. まともなシステムがパスワードを保存するとき、単純な高速ハッシュ一回では不十分なのはなぜか。ソルトとコストの役割に触れ、保存値が漏れても残るリスクを述べよ。
4. なりすましとリプレイをそれぞれ一文で説明せよ（一方は結果、一方は手法である点を踏まえて）。検証側が確かめるべき三つの性質（新しさ・文脈への束縛・正しい相手が作った裏付け）のうち、(a) 同一文脈での過去要求の再送を防ぐのはどれか、(b) 別のサービスや端末への流用を防ぐのはどれか。また、署名で「正しい相手が作った裏付け」を確かめていれば、それだけでリプレイを防げるか。
5. 「認証に成功した」という真偽値を `true` として送る方式には、性質の異なる二つの危険がある。それぞれ（リプレイと偽造）を説明し、「危ういのはブール型だから」ではないと言える理由を述べよ。
6. 脅威モデルで「守らないもの」を決めるとき、製品としての非ゴール・前提・範囲外の脅威・残余リスクは区別すべきである。bifrauth を例に、この四つをそれぞれ一つずつ挙げよ。また「ユーザーの誤承認」と「DoS」は、この四つのどれに分類されるか（正本 §4.2 に照らして）。
7. 「root を奪った攻撃者は bifrauth の防御対象外」とはどういうことか。verifier・PAM・登録情報が TCB にあることに触れて、なぜ保証が成り立たなくなるのかを説明せよ。
8. 「非特権プロセスが侵害されても単独では認証成功を偽造できない」を支える要素を、鍵の在りか・判断の場所・送られる証の性質の三点から説明せよ。またこの目標が「無条件の安全宣言」ではないことを、攻撃者に残る手段を一つ挙げて述べよ。
