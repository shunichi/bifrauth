## 3.1 PAM とは何か — なぜ差し替えられるのか

Linux では、本人確認を必要とするプログラムが数多くある。
コンソールログイン、`sudo`、画面ロックの解除、SSH、そして polkit（いずれも、その構成で PAM を利用するようビルド・設定されている場合）。
もしそれぞれが「パスワードをどう確かめるか」を自前で書いていたら、どうなるか。
二要素認証を足したいだけで、すべてのプログラムを書き換えて回ることになる。

そうならないための仕組みが **PAM**（Pluggable Authentication Modules、着脱可能な認証モジュール）である。
アプリケーションは「本人確認をしてほしい」とだけ PAM に頼み、実際にどうやって確かめるかは PAM の設定に任せる。
確かめ方は、`.so` という形の **モジュール**として差し込む。
だから「着脱可能」なのであり、パスワード、指紋、そして bifrauth の Face ID を、アプリを書き換えずに足したり外したりできる。

どのアプリがどう認証するかは、**サービス**という単位で設定される。
PAM を使うアプリはそれぞれサービス名を名乗る。
`login`、`sudo`、そして polkit なら `polkit-1` である。
このサービス名は、アプリが PAM を開始するときに自分で指定する値で、モジュールは PAM から `PAM_SERVICE` として受け取る。
設定は多くの Linux で `/etc/pam.d/<サービス名>` というファイルに書かれ、そのサービスがどのモジュールをどの順で使うかを決める（環境によっては単一の `/etc/pam.conf` を使うなど、置き場所には差がある）。
bifrauth の導入は、このサービスの設定へモジュールの行を加えることにあたる。
ただし、既存のシステム設定を自動で書き換えるのは危険なので、bifrauth のインストーラーは system の共通スタックを上書きせず、専用の設定と明示的な手順を用いる（正本設計書 §12.3。詳しくは §3.6）。

ここで、サービス名がどこから来るかに注意しておきたい。
サービス名は、カーネルが実行ファイルや操作を検証して割り当てる値ではない。
PAM を呼ぶアプリケーションが指定する、アプリ起点の値である。
だから bifrauth は、モジュール引数や環境変数、非特権な helper が自己申告する用途は信用せず、信頼できる呼び出し元（root 専用 IPC と `SO_PEERCRED` で確かめる。一般論は第 6 章、daemon 側の実装は第 19 章、PAM モジュール側は第 22 章）が渡す `PAM_SERVICE` を用途の根拠にする。
もっとも、サービス名だけでは、実際にどの実行ファイルやどの操作から来たかを暗号学的に証明できるわけではない。
とりわけ polkit は、どの操作（action）かを PAM へ伝えない。
この限界が bifrauth の用途の束縛にどう響くかは、第 4 章で立ち返る。

## 3.2 三者 — アプリケーション、モジュール、conversation

PAM のやり取りには、三つの役者が登場する。
役割を分けて見ておくと、後でどこが信頼できてどこが信頼できないかを判断しやすい。

一人めは **アプリケーション**である。
`login` や `sudo`、polkit の内部で PAM を呼ぶ `polkit-agent-helper-1` がこれにあたる。
アプリケーションは「このサービスで本人確認したい」と PAM ライブラリに依頼する側である。

二人めは **モジュール**である。
実際に「確かめる」処理を担う `.so` で、パスワードを照合するもの、bifrauth のように iPhone へ要求を出すもの、などがある。
アプリケーションはモジュールの中身を知らず、モジュールもどのアプリから呼ばれたかを直接は選べない。

三人めは、少し毛色が違う。
**conversation**（会話）は、モジュールがユーザーと言葉をやり取りするための通り道である。
モジュールが画面へ直接描くのではなく、「このメッセージを表示してほしい」「パスワードを尋ねてほしい」とアプリケーションへ頼み、アプリケーションが実際の入出力を行う。
つまり、ユーザーに何をどう見せるかを最終的に握るのはアプリケーション側である。

この最後の点は、後で重い意味を持つ。
モジュールが「メッセージを表示して」と頼み、成功が返ってきても、それはユーザーが実際にそれを目にした保証にはならない。
bifrauth が number matching の確認コードを表示するとき、この「表示できた ≠ 見られた」という隙間が問題になる（第 20 章）。
いまは、モジュールは自分で画面を描かず、アプリケーションを介してしか話せない、と押さえておけばよい。

## 3.3 四つの管理グループ

本人確認と一口に言っても、PAM はもう少し細かく仕事を分けている。
サービスの設定ファイルは、四つの **管理グループ**（management group）に分かれて書かれる。
それぞれが答える問いが違う。

- **auth**：あなたは誰か。本人であることの確認そのもの。パスワードや Face ID が働くのはここである。
- **account**：その本人に、今このアクセスを認めてよいか。アカウントが期限切れでないか、この時間帯・端末で許されているか、といった条件を見る。
- **password**：認証に使う秘密（パスワードなど）を変更するときの処理。
- **session**：認証が済んだ後、セッションを整える・後始末する処理。

bifrauth が主に住むのは auth である（正本設計書 §12.1）。
ただし、他の PAM サービスへ広げたときの挙動も定めておく必要がある。
そこで account については、状態を変えず成功を返すだけの no-op として振る舞う。
ここは誤解しやすい。
この no-op は「bifrauth のモジュールが account の観点で追加の拒否をしない」という意味であって、account スタック全体を許可する意味ではない。
最終的に account を通すかは、そのスタックに並ぶ他のモジュールとフラグが決める。
bifrauth は本人確認そのものだけを引き受け、アクセスの可否や期限といった判断には口を出さない、という役割の切り分けである。

## 3.4 コントロールフラグとスタックの合成

一つの管理グループには、モジュールを複数、順に並べられる。
この並びを **スタック**（stack）と呼ぶ。
そして、各行に付く **コントロールフラグ**（control flag）が、そのモジュールの結果を全体の結論へどう反映するかを決める。
ここが、PAM のいちばんの勘所である。

基本のフラグは四つある。
まず、成否がそのまま重みを持つものから。

- **required**：成功が必須。失敗しても即座には止めず、スタックの残りは評価するが、最終的な結論は失敗になる。どのモジュールで失敗したかを攻撃者に悟らせないための挙動である。
- **requisite**：required と同じく成功が必須。ただし失敗した瞬間にそこで打ち切り、残りを評価しない。

次に、途中で結論を確定させるものと、添え物として扱うもの。

- **sufficient**：このモジュールが成功すれば、それだけで全体を成功として打ち切れる。ただし条件があり、それより上に置かれた required なモジュールがすでに失敗している場合は、打ち切れない。失敗したときは、その結果は無視され、スタックの評価が続く。
- **optional**：結果は通常は無視される。ただし、そのグループに全体の結論を決める結果が他になかったときには、この結果が効くことがある。

大事なのは、スタックが上から下へ順に評価されること、そして順番と組み合わせで結論が変わることである。
「bifrauth を足しても既存のログインを壊さない」という設計は、まさにこの仕組みの上に成り立つ。
bifrauth を `sufficient` で、パスワード認証より上に置く。
Face ID が成功すればそこで認証は通り、失敗したり使えなかったりすれば、その結果は無視されて下のパスワードへ評価が進む。

なお、ここで挙げた四つのキーワードは、より細かく制御できる `[value=action ...]` という記法の、代表的な組み合わせに名前を付けたものである。
本書では基本のキーワードで話を進める。

## 3.5 モジュールの入口と戻り値

モジュールは `.so` だが、PAM から呼ばれる関数（入口）はあらかじめ決まっている。
管理グループに対応して、次のような入口がある。

- `pam_sm_authenticate` … auth（本人確認）
- `pam_sm_setcred` … auth に付随する資格情報の設定
- `pam_sm_acct_mgmt` … account
- `pam_sm_chauthtok` … password
- `pam_sm_open_session` / `pam_sm_close_session` … session

モジュールは、これらの入口で処理を行い、結果を **戻り値**として返す。
戻り値は単なる成功・失敗ではなく、何が起きたかを区別する。
bifrauth は、状況に応じて次のように返す（正本設計書 §12.2）。

- 認証成功：`PAM_SUCCESS`
- Face ID 拒否：`PAM_AUTH_ERR`
- iPhone に届かない、またはタイムアウト：`PAM_AUTHINFO_UNAVAIL`
- 対象ユーザーが不明：`PAM_USER_UNKNOWN`
- 内部エラー：`PAM_SYSTEM_ERR`

戻り値の区別には理由がある。
まず、成功と非成功を区別しないと危うい。
何でも「成功」にしてしまえば、次節のフォールバックが安全に働かなくなる。

そのうえで、非成功どうしの区別にも意味がある。
「Face ID を拒否された（`PAM_AUTH_ERR`）」は、本人確認が明示的に失敗した状態である。
「iPhone に届かなかった（`PAM_AUTHINFO_UNAVAIL`）」は、確認情報が得られず確認そのものを行えなかった状態である。
前節のような単純な `sufficient` 構成では、どちらも同じように下段のパスワードへ落ちる。
しかもその構成で下段が最終的に成功したなら、アプリケーションが上段モジュールの個別の戻り値をそのまま受け取るわけではない。
それでも区別が要るのは、落ちる先が同じでも、意味が違えばどこかで扱いが変わりうるからだ。
モジュール自身が残すログ、個別の戻り値が全体の結果へ反映されるようなスタックの組み方、あるいは `[value=action ...]` を使ったより細かいポリシーでは、「拒否された」と「届かなかった」を別々に扱える。
（ただし、失敗の詳細をそのままユーザーに見せることが常に良いとは限らない。攻撃者への手がかりになりうるからで、どこまで見せるかはアプリケーション側の判断になる。）

## 3.6 フォールバック — 失敗したらパスワードへ

ここまでの部品で、bifrauth の設定例を読めるようになった。
正本設計書 §12.3 は、初期導入時の auth スタックをこう示している。

```pam
auth sufficient pam_bifrauth.so timeout=30
auth include common-auth
```

一行めで bifrauth を `sufficient` として置き、二行めで既存のパスワード認証（`common-auth`）を取り込む。
Face ID が成功すれば、一行めで認証は通り、パスワードは尋ねられない。
Face ID を拒否されたとき、iPhone が見つからないとき、タイムアウトしたとき、あるいは verifier や helper が止まっているとき。
そのいずれでも、`sufficient` の失敗は無視され、評価は二行めのパスワードへ落ちる。

ただし、この「下へ落ちる」は無条件の魔法ではなく、いくつかの前提の上で成り立つ契約である。
`sufficient` が無視するのは、モジュールが**返してきた**非成功の結果である。
だから前提として、bifrauth のモジュールは、期限内にきちんと非 `PAM_SUCCESS` を返して戻ってこなければならない。
モジュールが戻ってこなかったり、呼び出し元のプロセスごと巻き添えにして落としたりすれば、`sufficient` は何も救えない。
だからこそ、モジュールは処理に期限を設け、内部の異常（panic）が呼び出し元へ伝播しないよう封じ込める（正本設計書 §19。この封じ込めは第 11 章・第 22 章で扱う）。
さらに、落ちる先である下段のスタックが正しく設定されている必要もある。
`common-auth` はディストリビューションが定義するスタックで、必ずしも単一のパスワードモジュールとは限らず、その下段自身の結果で最終的な可否が決まる。
だから bifrauth のインストーラーは、下段を勝手に推測して上書きしたりせず、既存の設定を尊重する（正本設計書 §12.3）。

これらの前提を満たしたうえでの「下へ落ちる」が、フォールバックの本体である。
逆に言えば、bifrauth のモジュールは、うまくいかなかったときに `PAM_SUCCESS` を返してはならない。
成功でないものを成功と偽れば、パスワードへ落ちる道が塞がれ、しかも認証が通ってしまう。
verifier の停止、通信エラー、明示的な拒否のどれであっても、成功を返さず、無期限にブロックもしない（正本設計書 §19）。

ここで注意したいのは、`sufficient` の順序が意味を持つことである。
もし bifrauth より上に、失敗している required なモジュールがあれば、bifrauth が成功しても全体は通らない。
設定を読むときは、フラグだけでなく並び順まで見る必要がある。

フォールバックが保証するのは「Face ID が使えなくてもログインできる」ことであって、「Face ID をかいくぐって誰でも入れる」ことではない。
下段のパスワード認証は、依然として自前の強さで守っている。
bifrauth は、その上に本人確認の別経路を一つ足しているだけである。

次の章では、この PAM の上に polkit がどう乗るのかを見る。
とりわけ、polkit がサービス名として一律に `polkit-1` を使い、どの操作かを PAM へ伝えないという制約が、bifrauth の用途の束縛にどう響くのかを追う。

## 理解度クイズ

> 解答と解説は付録 B にまとめてある。まず自分で考えてから確認してほしい。

1. PAM が存在することで、新しい認証方式（たとえば Face ID）を Linux に足すとき、何をせずに済むか。PAM がなかった場合と比べて説明せよ。
2. PAM のサービス名は誰が指定する値か（カーネルが実行ファイルを検証して付ける値か、それともアプリ起点の値か）。bifrauth がモジュール引数や非特権 helper の自己申告ではなく `PAM_SERVICE` を用途の根拠にするのはなぜか。またサービス名だけでは証明できないことは何か。
3. PAM の三者（アプリケーション・モジュール・conversation）をそれぞれ一言で説明せよ。モジュールはユーザーの画面へ直接描けるか。
4. conversation が「メッセージを表示して」と頼んで成功が返っても、それだけでは保証できないことは何か。bifrauth のどの機能でこの隙間が問題になるか。
5. auth と account は、それぞれどんな問いに答える管理グループか。bifrauth が主に住むのはどちらで、もう一方に対してはどう振る舞うか。またその no-op（成功を返す）は「account スタック全体を許可する」という意味か。
6. `PAM_AUTH_ERR`（明示的な拒否）と `PAM_AUTHINFO_UNAVAIL`（確認情報を得られない）は、どう意味が違うか。単純な `sufficient` 構成では両者とも下段へ落ちるのに、区別する意味はどこにあるか。
7. コントロールフラグ `required` と `requisite` は、失敗したときの挙動がどう違うか。
8. `sufficient` なモジュールが成功したとき、常に全体が成功で打ち切られるとは限らない。打ち切れないのはどんな場合か。
9. 次の設定で、Face ID（`pam_bifrauth.so`）が iPhone 未到達で失敗したとき、認証全体はどうなるか。またこの「下段へ落ちる」フォールバックが成り立つために、モジュールと下段スタックが満たすべき前提を二つ挙げよ。
   ```pam
   auth sufficient pam_bifrauth.so timeout=30
   auth include common-auth
   ```
10. bifrauth のモジュールが、通信エラーや verifier 停止のときに `PAM_SUCCESS` を返してはならないのはなぜか。また `sufficient` は「返された非成功」を無視する仕組みである。モジュールが結果を返さず呼び出し元プロセスごと落ちてしまった場合、`sufficient` はフォールバックを救えるか。
