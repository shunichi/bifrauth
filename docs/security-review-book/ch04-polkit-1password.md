# 第 4 章 polkit と 1Password の system authentication

> **この章の実装状態:** polkit 自体は一般知識として扱う。bifrauth 固有の記述には節ごとにラベルを付ける。
> polkit 統合は〔仕様（正本）§9.2 / §9.3 / §12.4 / §21。初期スコープでは polkit・1Password 連携は
> Mac/実機が揃う段階の対象で、**Linux 側の実機統合・実機確認はいずれも未実施**（progress.md の項目
> 14〜16 は未着手）。参照時点: commit `7789033`（2026-07-19 JST）〕。

bifrauth の主用途は、1Password を Face ID で解錠することである。
「1Password だけを Face ID で開く」と言われると、いかにも狙いを絞った話に聞こえる。
だが前章の終わりで触れたとおり、PAM が受け取るサービス名は `polkit-1` という一つの名前でしかなく、それがどの操作から来たのかまでは分からない。
すると、こういう疑問が湧く。
Face ID で開けるのは本当に 1Password だけなのか。
それとも、同じ `polkit-1` を通る他の操作も、いっしょに開けてしまうのか。

この章は、その疑問に正面から答えるための下ごしらえである。
まず polkit が何をする仕組みかを見て、認証エージェントと `polkit-agent-helper-1` の役割を追う。
そのうえで、action の識別子がどこまで届き、どこで失われるのかを確かめる。
最後に、1Password の system authentication で「設計書が前提として記述していること」と「このプロジェクトで実際に確かめた証跡」を分ける。

---

## 4.1 polkit とは — 操作ごとの認可の調停役

第 2 章で、認証と認可を分けた。
polkit は、そのうちの **認可**を引き受ける仕組みである。
より正確には、「ある主体（subject、要求してきたプロセス）が、ある操作を実行してよいか」をシステム全体で判断する調停役である。

polkit が扱う一つひとつの操作を **action** と呼ぶ。
action には識別子（action ID）が付いていて、たとえばプリンタの設定変更や、ソフトウェアのインストール、そして 1Password の解錠が、それぞれ別の action として定義される。
action ごとに、既定でどう扱うかが決められている。
無条件に許す、無条件に拒否する、本人の認証を求める（auth_self）、管理者の認証を求める（auth_admin）、といった結果である。

ここで、主体と認証される identity を混同しないでおきたい（第 2 章の区別を保つ）。
auth_self は、要求してきたプロセスの Unix ユーザーそのものというより、セッションの持ち主にあたる identity を認証する、という意味である。
auth_admin は、あらかじめ設定された管理者の identity を認証する。
どちらも「要求してきたプロセス＝認証される相手」とは限らない。

もう一つ、見落とすと後で誤解する要素がある。
これらの結果には、認証をしばらく保持する変種（auth_self_keep、auth_admin_keep）がある。
同じ action ID と主体の組に対して一時的な認可（temporary authorization）が保持されると、その間の判定は、認証エージェントや PAM を再び呼ばずに「許可」となることがある。
つまり、system authentication を要求するたびに必ず Face ID の問いかけが走る、とは限らない。
これは polkit が認可の結果をしばらく保持する性質であって（単に PAM の成功結果をキャッシュするのとは違う）、bifrauth の設計上の保証や UX と混同してはいけない。
どの結果が返り、認可が保持されるかは、対象環境のポリシー次第である（§4.4 で確認事項として扱う）。

polkit の第一の仕事は、あくまで「許すか許さないか」の判断である。
本人確認そのものは polkit の仕事ではない。
action が認証を求めるとき、polkit はその本人確認を、次に見る認証エージェントと PAM へ委ねる。

## 4.2 認証エージェントと polkit-agent-helper-1

ある action が本人確認を必要とするとき、polkit は誰かにユーザーへ尋ねてもらわなければならない。
その役を担うのが **認証エージェント**（authentication agent）である。
デスクトップ環境が用意する部品で、ユーザーのセッションの中で動き、認証を促すダイアログを画面に出す。

このとき polkit は、認証エージェントに対しては、どの action かを含む情報（action ID、表示用のメッセージや詳細など）を渡す。
だから認証エージェントは、「これは何の操作の認証か」をユーザーへ説明できる。
既存のエージェントが表示するこの説明（intent 表示）は、bifrauth でも維持される（正本設計書 §9.3）。

ただし、認証エージェントはユーザー権限で動く。
第 1 章で見たとおり、ユーザー権限のプロセスは信頼の置きどころとして弱い。
そこで、実際に本人確認の判定へ進むところは、別の特権を持つ部品が担う。
対象とする Linux とエージェントでは、これは `polkit-agent-helper-1` という小さな setuid ヘルパーで、ここから PAM がサービス名 `polkit-1` で起動される。

ここは一般化しすぎないよう注意が要る。
polkit のエージェント用ライブラリは、いまのところ内部の setuid ヘルパーを使ってネイティブ認証（たとえば PAM）へ接続する、という作りになっている。
だが、どのヘルパーを経てどの PAM スタックを通るかは、エージェントの実装、polkit の版、ディストリビューションに依存する実装上の経路である。
すべてのエージェントが必ず同じヘルパーと PAM を使うとは限らない。
だから「対象環境で実際に `polkit-1` の PAM スタックを通るか」も、実機で確かめるべき事柄になる（§4.4）。

対象環境での一連の流れを、登場順に並べておく。

1. 1Password などが、ある action の実行を polkit へ求める。
2. polkit が、その action は本人確認が要ると判断する。
3. ユーザーセッションの認証エージェントが、action の説明を含むダイアログを出す（ここには action ID が届いている）。
4. `polkit-agent-helper-1` が PAM をサービス `polkit-1` で起動する（ここで action ID は引き継がれない）。
5. `polkit-1` の PAM スタックに置かれたモジュール（ここに bifrauth を入れうる）が本人確認を行う。

bifrauth は、この 5 の位置に入る。

## 4.3 action ID はどこまで届き、どこで失われるか

さて、先ほどの疑問に戻る。
Face ID で開けるのは 1Password だけか、それとも `polkit-1` を通る他の操作もか。

鍵は、action ID がどこまで届くかにある。
前節で見たとおり、action ID は認証エージェントには届く（だからエージェントは操作を説明できる）。
それが失われるのは、その先である。
ヘルパーが PAM をサービス `polkit-1` で起動する境界を越えると、action ID は引き継がれない。
PAM モジュールも、その先の bifrauth の verifier も、「これは 1Password の解錠だ」「これは別の管理操作だ」を区別できない。
どちらも同じ `polkit-1` の本人確認として、まったく同じに見える。

この対比が、設計の一つの帰結を生む。
エージェントの intent 表示は維持できる（action ID がそこには届くから）。
しかし、その表示内容を暗号学的な判定や署名対象には使えない（PAM から先には action ID が届かず、そもそも非特権なエージェントの表示を信頼の根拠にできないから。正本設計書 §9.3）。

この事実から、bifrauth のスコープが決まる。
〔仕様（正本）§9.2 / §12.4〕
bifrauth は、PAM サービス名から導いた用途（たとえば `polkit-1.authenticate`）へは、要求を暗号学的に束縛できる。
これによって、polkit 用の認証を `sudo` や `login` など別の PAM サービスへ流用する攻撃は防げる。
しかし、同じ `polkit-1` の中では、1Password の action と他の action を区別できない。
だから、`pam_bifrauth` を `polkit-1` の認証スタックへ入れると、そのスタックで本人確認を行う他の polkit action にも Face ID が使えるようになる。

用途を後から名乗らせて絞れないか、と考えたくなる。
理由を分けて見ておく。
非特権なエージェントが実行時に「これは `1password.unlock` だ」と申告する方式は、その申告者自身を認証できないので採れない。
一方、root が管理する PAM 設定に静的に用途を書き込むのは、攻撃者の自己申告とは違う。
だがそれでも解決にはならない。
`polkit-1` の行へ静的に `1password.unlock` と書いても、同じスタックを通るすべての action を一律に「1Password だ」と誤ってラベル付けするだけで、実際にどの action から来たかを証明したことにはならないからだ。
どちらの道も action ID の証明にはならず、設計はこの用途申告方式を採らない（正本設計書 §12.4）。

では polkit の側で action ごとに絞れないのか。
polkit には `rules.d` という仕組みがあり、action ID ごとに「許す・拒否する・本人認証・管理者認証」を返せる。
これを使えば、ある action だけ認証を要求し、他を無条件に許可または拒否する、という制御はできる。
できないのは、その先である。
認証を要求する複数の action の間で、action ごとに別の PAM サービスやスタックやモジュールを選ぶことは、この仕組みではできない。
auth_admin にしても、認証される identity が管理者になるだけで、別の PAM スタックへ移るわけではなく、依然として `polkit-1` を共有する（正本設計書 §12.4）。
だから「この action だけ Face ID、他は従来の認証」という振り分けはできない。
レビューでは、bifrauth を `polkit-1` へ入れた環境で、同じスタックを共有する action が他に何かを必ず確かめることになる。

## 4.4 1Password の system authentication と、確かめるべきこと

1Password の「Unlock using system authentication」を有効にすると、解錠の要求が polkit を経由する。
公式の説明では、1Password は polkit の action ポリシーと `polkit-1` の PAM を利用する。
そこから先は、前節までの流れに乗る。
polkit、認証エージェント、`polkit-agent-helper-1`、PAM（`polkit-1`）、そして `pam_bifrauth` へと届く。
1Password 自体は改変せず、polkit と PAM を通じて bifrauth を利用する、という構図である。

ただし、ここで断定を避けるべきことが二つある。
一つは、パッケージ形態や版によっては、この system authentication に対応しないことがある点である。
たとえば Snap や Flatpak といった隔離された形式では、前提が変わりうる。
どの版がどう動くかは、環境ごとに違う。

もう一つ、より重要な区別がある。
「設計書が前提として記述していること」と、「このプロジェクトで実際に確かめた証跡」は、別物である。
正本設計書 §3 は、対象の polkit 認証エージェントが、action の説明と PAM のメッセージ（`PAM_TEXT_INFO`）を同じダイアログに表示することを、**満たされているべき前提（受入条件）**として記述している。
これは「満たされていなければならない条件」であって、「すでに満たされていることを確認した」という報告ではない。
実際、このプロジェクトでは対象エージェントや版でのこの確認はまだ行われていない（progress.md の項目 14 は未着手）。
したがって現時点では、この前提は未検証である。
そして設計は、この確認が取れるまで polkit への統合を有効化・サポートしない、という立場を取る（正本設計書 §9.3 / §19）。

だからレビューでは、次のようなことを実機確認の対象として扱う。
対象の 1Password のどの版が、どの action ID で、どんな流れを起動するか。
どの認証エージェントが使われ、それが action の説明と PAM のメッセージを同じダイアログに、認証開始前に表示するか。
実際に `polkit-1` の PAM スタックを通るか。
そのポリシーがどの既定結果を返し、一時的な認可（keep）を保持するか。
これらを「一般に必ずこうなる」と断定せず、対象環境で確かめた結果として扱う。

この区別は、セキュリティレビューの姿勢そのものでもある。
「設計書がこう書いている」ことと、「この環境で実際にこう動く」ことは別である。
前者は前提や受入条件として読み、後者は実機確認の対象として扱う。
そして未確認のことを、確認済みであるかのように保証に数えない。

次の章では、視点をがらりと変える。
ここまでは Linux 側の枠組み（PAM と polkit）を見てきた。
次は、その枠組みの中で「本人であること」を実際に支える暗号の部品へ移る。
署名、ハッシュ、その場かぎりの問いかけ、そして Face ID を鍵のゲートとして使う、という核心である。

## 理解度クイズ

> 解答と解説は付録 B にまとめてある。まず自分で考えてから確認してほしい。

1. polkit は、第 2 章で分けた認証と認可のどちらを主に引き受ける仕組みか。一言で言うと何を判断するか。
2. polkit の action とは何か。action ごとに既定で決められている結果を、二つ以上挙げよ。auth_self が認証するのは「要求してきたプロセスの Unix ユーザーそのもの」だと言い切ってよいか。
3. auth_self_keep や auth_admin_keep のような「保持」の結果があると、system authentication を要求するたびに必ず Face ID の問いかけが走る、とは限らない。なぜか。これは bifrauth の設計上の保証と同じものか。
4. ある action が本人確認を必要とするとき、ユーザーへ尋ねる役を担うのは何か。それはどの権限で動くか。実際にどのヘルパーを経てどの PAM スタックを通るかは、常に同じと断定してよいか。
5. polkit が渡す action ID は、認証エージェントには届くか。PAM モジュールや verifier には届くか。両者を区別して答え、それが「エージェントの intent 表示は維持するが署名対象には使えない」という結論にどうつながるかを述べよ。
6. bifrauth が用途の自己申告方式を採らない理由を、(a) 非特権エージェントが実行時に申告する場合、(b) root 管理の PAM 設定へ静的に `1password.unlock` と書く場合、の二つに分けて説明せよ。
7. polkit の `rules.d` を使ってできること（ある action だけ認証を要求し、他を許可/拒否する）と、できないこと（認証を要求する action の間で action ごとに別の PAM スタックを選ぶ）を、それぞれ述べよ。auth_admin にすると別の PAM スタックへ移るか。
8. `pam_bifrauth` を `polkit-1` の認証スタックへ入れるとき、レビューで必ず確かめるべきことは何か。
9. 正本設計書 §3 が「エージェントが action の説明と PAM メッセージを同一ダイアログへ表示する」と記述していることは、「このプロジェクトでそれが実機確認済みである」ことを意味するか。現時点の状態（progress.md 項目 14）に触れて答えよ。
