# 第 5 章 暗号の最小限と Secure Enclave

> **この章の実装状態:** 暗号の一般知識を扱う。bifrauth 固有の記述には節ごとにラベルを付ける。
> Ed25519 署名・検証と P-256 検証は〔実装済み。`bifrauth-crypto`。参照時点: commit `42e9118`
> （2026-07-19 JST）〕。iPhone 側の Secure Enclave 署名は〔仕様（正本）§8.1 / §9.6。実機は未実装で、
> このマシンでは `mock-iphone` がソフトウェアの P-256 鍵で代替（第 21 章）〕。

第 2 章で、こう約束した。
使い回せる秘密も、成功を表す真偽値も送らない。
代わりに、その場かぎりの問いかけに、鍵でしか作れない答えを返させる。
だが、そのときは仕掛けの名前（チャレンジレスポンスと署名）を出しただけで、中身には踏み込まなかった。

この章で、その中身を開ける。
公開鍵暗号の署名とは何か、そして検証が本当に保証するのはどこまでか。
ハッシュは何のためにあり、何を仮定しているのか。
「その場かぎり」と「この文脈のため」を、署名でどう表すのか。
そして、iPhone の Secure Enclave と Face ID が、この仕組みのどこに、どんな条件つきで収まるのか。
最後にもう一度、なぜ真偽値を送ってはいけないのかへ戻る。

## 5.1 公開鍵暗号と署名 — 検証が保証するのはどこまでか

暗号の道具のうち、この本で要るのは主に一つ、**署名**（signature）である。

署名は、**鍵ペア**（key pair）を使う。
対になった二つの鍵があり、一方を **秘密鍵**（private key）、もう一方を **公開鍵**（public key）と呼ぶ。
秘密鍵は持ち主だけが握り、公開鍵は誰に見せてもよい。
あるデータに対して秘密鍵で署名を作り、その署名は対応する公開鍵で検証できる。

ここで、署名の検証が「何を」保証するのかを、控えめに正確に述べておきたい。
検証が通ったとき、単独で分かるのは次のことだけである。
その公開鍵に対応する秘密鍵によって作られた有効な署名であること。
そして、署名の対象になったデータが、署名の後で変えられていないこと。

つまり、検証だけでは「誰が」署名したかまでは分からない。
「その公開鍵が確かにあの iPhone のものだ」という結び付きは、検証が与えるのではなく、安全な登録（ペアリングとレジストリ。第 10 章・第 18 章）が与える。
だから、もし攻撃者が登録済みの公開鍵を自分の鍵へ差し替えられたなら、攻撃者の署名も「有効」として通ってしまう。
検証の結論は、いくつかの前提の上に立っている。
公開鍵と端末の結び付きが信頼できること、秘密鍵が漏れていないこと、署名方式そのものが破られていないこと。
これらが崩れれば、検証が通ることは本人性を意味しなくなる。

そのうえで、一つの誤解を払っておく。
署名は「秘密鍵でデータを暗号化すること」ではない。
暗号化は中身を隠す操作で、署名は「対象のデータが、この秘密鍵と結び付いた形で作られ、その後変えられていない」ことを示す操作である。
公開鍵を持つ者は署名を検証できるが、新しい署名を偽造はできない。
偽造には秘密鍵が要り、それは持ち主の手元だけにあるからだ。
この非対称のおかげで、検証する側（Linux）は相手の公開鍵さえ持てばよく、公開鍵が漏れてもそれだけでは署名を偽造されない。

## 5.2 ハッシュ — 何に署名し、何を仮定しているのか

署名の内部では、対象データそのものより、その **ハッシュ**（hash）が使われることが多い。

ハッシュ関数は、任意の長さのデータを受け取り、決まった長さの短い値へ変換する。
本書で使うのは **SHA-256** で、出力は 256 ビット（32 バイト）に固定される。
暗号学的なハッシュに求められる性質は、次のように整理できる。

- **原像計算困難性**（preimage resistance）：ハッシュ値から、それを生む入力を見つけるのが計算量的に困難。
- **第二原像計算困難性**（second-preimage resistance）：ある入力が与えられたとき、同じハッシュ値になる別の入力を見つけるのが困難。
- **衝突困難性**（collision resistance）：同じハッシュ値になる二つの入力の組を、どれでもよいから見つけるのが困難。
- **なだれ効果**（avalanche）：入力をわずかに変えると、出力のビットが大きく変わるように設計されていること。

ここで正確を期したい。
出力は 256 ビットに固定されているので、理論上は、異なる入力が同じハッシュ値になる組（衝突）が必ず存在する。
暗号学的ハッシュが与えるのは「衝突が存在しない」ことではなく、「衝突を現実的に見つけられない」ことである。
だからハッシュ値の一致は、同一性の数学的な証明ではなく、衝突が現実には見つからないという前提の下での比較である。
SHA-256 について「現実的に困難」と「不可能」は分けて考える。

もう一つ、後の章の要になる点を、bifrauth の契約として今ここで明示しておく。
署名する側と検証する側は、**まったく同じバイト列**を対象にしなければならない。
一文字でも並び順でも違えば、対象が変わり、検証は通らない。
そのため bifrauth では、署名の対象を、実装が違っても必ず同じバイト列になるよう厳密に決める（正規化。一般論は第 7 章、実装は第 13 章）。
さらに、この「何を渡すか」には具体的な決まりがある。

- Ed25519 の API へは、正規化済みの問いかけ（canonical challenge）の生のバイト列全体を、そのまま渡す。
- iPhone 側の ECDSA では、未加工の canonical bytes 全体を「メッセージ用」の署名 API（`ecdsaSignatureMessageX962SHA256`）へ渡し、SHA-256 は Security.framework がちょうど一度だけ行う。あらかじめ計算した 32 バイトのハッシュを、このメッセージ用 API へ渡してはならない。
- Rust 側の検証も、同じ canonical bytes をメッセージ用 API へ渡す。

なぜこの区別が要るのかというと、ハッシュを二重に掛けてしまう罠があるからだ。
その詳細は実装の章（第 14 章）で扱うが、入力の契約だけはこの章に置いておく。

## 5.3 nonce とチャレンジレスポンス — 「その場かぎり」をどう作るか

署名だけでは、まだ足りない。
正しい署名でも、それが過去のものなら、再送で通ってしまう（第 2 章のリプレイ）。
これを防ぐのが、**チャレンジレスポンス**（challenge-response）である。

流れはこうである。
検証する側が、毎回新しく作った問いかけ（チャレンジ）を相手へ渡す。
相手は、その問いかけに署名して答え（レスポンス）を返す。
検証する側は、それが自分が今出した問いかけへの署名かどうかを確かめる。

問いかけを毎回新しくするために、**nonce**（number used once、一度だけ使う数）を使う。
nonce は、推測できないよう乱数で作り、要求ごとに作り直す。
ただし、ここが誤解しやすい。
メッセージに nonce が入っているだけ、あるいは毎回乱数を作るだけでは、古い答えを拒否できない。
「新しさ」を保証するのは、検証する側の状態管理のほうである。

つまり、検証する側は次をすべて行わなければならない。
自分が今発行して保留（pending）に持っている nonce や request ID と一致するかを確かめる。
有効期限を過ぎていないかを確かめる。
そして、一度使った要求を、その場で一度だけ消費（consume）して、二度と通さない。
この消費は、検証の開始時に行う。
署名が不正だった場合も含めて request ID を消費するので、攻撃者が同じ要求で何度も試し直すことはできない（正本設計書 §9.7）。
これらがそろって、はじめて再送が弾ける。

〔仕様（正本）§16 / §9.7〕
bifrauth では、nonce と request ID を、それぞれ独立に乱数で 128 ビット以上の値として作り、一度答えを受けた要求は使い切りにする。
なお、問いかけに含まれる発行時刻や有効期限に署名がついていても、それだけで Linux 側の期限判定にはならない。
期限の権威は、verifier が別の時計（`CLOCK_BOOTTIME`）で測る保留状態のほうである（詳細は第 8 章・第 16 章）。

## 5.4 文脈の束縛 — 誰が・どのサービスで・いつ・どの端末で

第 2 章で、もう一つの性質を挙げた。
文脈への束縛である。
新しさだけでは、ある場面向けの正しい答えを、別の場面へ流用されうる。

署名は、この束縛をきれいに表せる。
署名の対象に、文脈そのものを含めてしまえばよい。
誰のための要求か（対象ユーザー）。
どのサービスから来たか（PAM サービス名から導いた用途）。
いつのものか（発行時刻と有効期限）。
どの端末に向けたものか（Linux と iPhone の識別子）。
これらを署名の対象に含めれば、署名はその文脈へ縛りつけられる。

すると、別の文脈へ持っていっても通らない。
`polkit-1` 向けの答えを `sudo` 向けに使おうとしても、署名対象に含まれる用途が違うので検証が落ちる。
端末 A 向けの答えを端末 B へ流しても、同じく落ちる。
第 2 章で「別文脈への流用を防ぐのは文脈への束縛だ」と言葉で述べた。
その束縛の正体が、署名対象へ文脈を含めることである。

## 5.5 本書の二つの署名 — Ed25519 と ECDSA P-256

bifrauth には、役割の違う署名が二つ登場する。
どちらも楕円曲線を使う署名方式だが、誰が、何のために署名するかが違う。
二つに分かれているのは、暗号強度を競わせた結果ではなく、動く場所と役割による選択である（どちらも中身を隠す機能＝機密性は持たない）。

一つめは **Ed25519** である。
〔仕様（正本）§9.1。検証・署名の実装は `bifrauth-crypto`〕
Linux 側の verifier（root で動く判断役）が持つ長期の鍵で、問いかけ（canonical challenge）そのものに署名する。
iPhone は、登録済みの verifier の公開鍵でこの署名を検証し、「この問いかけは、確かに登録済みの Linux から来た本物だ」と確かめる。
Ed25519 署名は、問いかけの出所と完全性を保証する。

二つめは **ECDSA P-256**（P-256 という楕円曲線を使う ECDSA）である。
〔仕様（正本）§9.6。iPhone 実機は未実装、`mock-iphone` が代替〕
iPhone 側が持つ鍵で署名し、Linux 側の verifier が登録済みの iPhone の公開鍵で検証する。

ここで、何に署名しているのかを正確にしておきたい。
iPhone が署名するのは、返信メッセージ（応答）の一式そのものではない。
署名の対象は、あくまで保存済みの canonical challenge のバイト列全体である。
応答には、request ID や iPhone の端末 ID、`signed_payload_hash`（＝ `SHA-256(canonical_challenge)`）といった項目も入るが、これらをまとめて署名しているわけではない。
`signed_payload_hash` は照合を助ける補助的な値にすぎず、認証の入力としては信用しない。
verifier は、保留中の要求の canonical bytes から自分でハッシュを計算し直して照合し、署名検証もその canonical bytes に対して行う。
応答に書かれた iPhone 端末 ID も、どの登録鍵で検証するかのヒントにすぎず、本人性の根拠は「選んだ登録鍵での署名検証が通ったこと」である。

二つを並べると、往復の全体が見えてくる。
Ed25519 が、問いかけが本物であることを iPhone に保証し、ECDSA P-256 が、答えが本物であることを Linux に保証する。
どちらも、文脈を含む同じ問いかけ（canonical challenge）を対象にする。
（この二つは、片方がもう片方の署名を署名し合う「鎖」ではない。同じ canonical challenge に対して、それぞれが別々に署名しているだけである。）
この往復の二つの署名のおかげで、通信路そのものを信用しなくても、問いかけと答えの出所・完全性は両端で確かめられる。
ただし、これが保証するのは出所と完全性であって、それ以上ではない。
署名は、通信の中身を隠すこと（機密性）も、通信量の観察や妨害（DoS、正規メッセージの握りつぶし）を防ぐことも、しない。
盗聴防止や非ペア端末の抑止は、相互認証された暗号化チャネルという別の層が担うが、その方式は未決で実装もこれからである（第 10 章・第 23 章）。

（なお、ここでいう「問いかけの nonce」は、プロトコル上の nonce であって、ECDSA の署名生成が内部で使う署名ごとの秘密値（一般に ECDSA nonce と呼ぶ）とは別物である。後者は乱数で作ることも決定的に作ることもでき、生成方法は Security.framework などの実装に委ねられる。）

## 5.6 Secure Enclave と、Face ID をゲートにする

では、iPhone の ECDSA P-256 の秘密鍵は、どこにあるのか。
ここで **Secure Enclave** が登場する。

Secure Enclave は、iPhone のハードウェアの中にある、鍵を守るための独立した領域である。
そこで作られた秘密鍵は、原則として外へ取り出せない。
アプリからは「この鍵で署名して」と依頼できるが、鍵そのものの中身は読めない。
本書では、この「取り出せない」を前提として扱う（第 2 章の分類でいう assumption。正本設計書 §3。iPhone OS や Secure Enclave の完全な侵害は §4.2 で範囲外）。

そして、ここに Face ID が結び付く。
Face ID は、認証の結果として使うのではない。
Secure Enclave の秘密鍵を「今、使ってよい」と許可するゲートとして使う。

ただし、「使うたびに Face ID が要る」「Face ID でしか使えない鍵」という言い方は、条件つきで受け取る必要がある。
これらは、鍵を正しい属性で作り、正しく運用したときに成り立つ保証である。
〔仕様（正本）§8.1〕
bifrauth の設計は、そのために次を求める。
鍵を Secure Enclave 内に、署名専用（`privateKeyUsage`）で作る。
生体登録のセットが変わったら鍵を無効にする設定（`.biometryCurrentSet`。iPhone の生体登録をやり直すと、その鍵は使えなくなる）。
署名のたびに新しい認証コンテキストを使い、過去の Face ID 成功を長く再利用しない。
アプリがバックグラウンドへ回った後の未処理要求を無効にする。
これらが正しく実装・運用されて、はじめて「Face ID でしか使えない」が成り立つ。
そして実機の iPhone アプリはまだ実装されておらず、このマシンでは `mock-iphone` が代替している点も忘れない。

もう一つ、Face ID が保証する範囲にも線を引く。
Face ID の成功は、その鍵を使ってよいと認可するだけである。
ユーザーが「何を承認したつもりか」という意図までは証明しないし、Linux 側のどの action かも当然分からない（第 4 章）。
だから bifrauth は、Face ID の低摩擦化による反射的な承認を、別の仕組み（number matching。第 20 章）で補おうとする。

## 5.7 なぜ真偽値を送ってはいけないのか — 回収

ここまでの言葉で、第 2 章の問いへ戻れる。
なぜ `{ "face_id": true }` を送ってはいけないのか。

危ういのは、値がブール型だからではない。
危ういのは、Face ID が成功したという裸の主張だけを、鍵による証明も、その場かぎりの値も、文脈への束縛もないまま信じてしまうことである。
そういう主張には、二つの弱点がある。
一つ、それは秘密鍵なしに誰でも作れる（偽造）。
二つ、一度通ったものを記録して再送できる（リプレイ）。

署名は、この二つをちょうど塞ぐ。
その場かぎりの nonce を含み、文脈に束縛された問いかけへの署名は、まず偽造できない。
作るには、Secure Enclave の中にある、Face ID でしか使えない秘密鍵が要るからだ。
そして再送もできない。
検証する側が、今保留している問いかけとの一致・期限・一度きりの消費を確かめるからだ（§5.3）。

裏を返せば、署名や MAC を付け、その場かぎりの値や宛先に束縛したうえでなら、Face ID の成功を表すトークンを設計することも一般には可能である。
問題は「真偽値」そのものではなく、裏付けのない裸の主張を信じることにある。
bifrauth が選んだのは、Face ID の成否という主張を運ぶのではなく、Face ID をゲートとして作られた canonical challenge への署名を検証する、という設計である。
Linux 側はその署名を検証し、偽造でも再送でもないことを確かめて、はじめて本人と認める。

もっとも、この仕組みが働くには、前提がいくつも要る。
検証に使う公開鍵が、攻撃者に差し替えられていないこと（§5.1）。
verifier の秘密鍵が、非特権プロセスから読めないこと。
保留状態の管理が正しく、一度きりの消費が守られること。
これらは第 1 章の「root だけが書ける・読める」と、その安全な守り方につながる。
次の章では、こうした鍵やソケットを、攻撃者に触らせずに置き、安全に開く方法を見る。
ローカルの通信とファイル、そして永続化の安全性である。

## 理解度クイズ

> 解答と解説は付録 B にまとめてある。まず自分で考えてから確認してほしい。

1. 署名の検証が通ったとき、それ「単独で」分かることは何か。逆に、検証だけでは分からず、別の仕組み（ペアリングや登録）が与えるものは何か。
2. 「署名とは、秘密鍵でデータを暗号化することだ」という説明はなぜ不正確か。また、攻撃者が登録済みの公開鍵を自分の鍵へ差し替えられた場合、検証は本人性を意味するか。
3. 暗号学的ハッシュに求められる耐性を三つ挙げよ。256 ビット出力のハッシュに「衝突は存在しない」と言えるか。言えないなら、暗号学的ハッシュが実際に与えているのは何か。
4. 「メッセージに nonce が入っていて、毎回乱数で作り直す」だけでは、古い答えの再送を拒否できない。再送を拒否するために、検証する側が状態として行うべきことを三つ挙げよ。
5. 署名対象に「対象ユーザー・用途（PAM サービス由来）・有効期限・端末識別子」を含めると、どんな攻撃を防げるか。`polkit-1` 向けの答えを `sudo` 向けに流用する例で説明せよ。
6. iPhone の ECDSA P-256 が署名する対象は、応答メッセージ一式か、それとも別のものか。応答に含まれる `signed_payload_hash` や iPhone 端末 ID を、verifier はどう扱うか。
7. Secure Enclave の秘密鍵について本書が置いている前提は何か（第 2 章の assumption / out-of-scope の区別に触れて）。「Face ID でしか使えない鍵」という保証は、無条件に成り立つか。
8. Face ID は「認証の結果」としてではなく何として使われるか。Face ID の成功が保証しないこと（ユーザーの意図や Linux の action）に触れよ。
9. `{ "face_id": true }` のような裸の主張を信じる方式に欠けている二つの性質を挙げ、署名がそれぞれをどう塞ぐかを説明せよ。「危ういのはブール型だからではない」と言えるのはなぜか。
