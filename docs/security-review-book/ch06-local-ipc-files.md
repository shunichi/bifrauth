# 第 6 章 ローカル通信とファイル・永続化の安全性

> **この章の実装状態:** OS の一般知識を扱う。bifrauth 固有の記述には節ごとにラベルを付ける。
> ローカル IPC（Unix ソケット・`SO_PEERCRED`）は〔実装済み。`bifrauth-ipc` / `bifrauthd`〕、
> デバイスレジストリのファイル安全性・安全な読み書きは〔実装済み。`bifrauthd/src/registry.rs`〕。
> ただし永続化の crash consistency は**一部のみ実装済み**（§6.4 で範囲を明示する）。
> 実装状態を確認した参照時点は commit `5252717`（2026-07-20 JST）で、これは本章ファイル自体の
> commit ではなく「この時点のコードで実装状態を判定した」という基準を指す。

第 5 章の終わりで、宿題を三つ残した。
検証に使う公開鍵が、攻撃者に差し替えられていないこと。
verifier の秘密鍵が、非特権プロセスから読めないこと。
保留状態の管理が正しく、一度きりの消費が守られること。
どれも「攻撃者に触らせない場所へ置き、安全に読み書きする」という一点に帰る。

第 1 章で、その土台は引いた。
root だけが書けるファイル、という壁である。
だが同じ第 1 章で、その壁だけでは足りないことも見た。
親ディレクトリを書けたら名前ごと差し替えられるし、開く途中で symlink をすり替えられるかもしれない。

この章では、その「足りない分」を実務のレベルまで具体化する。
まず、root だけが話せる通信路をどう作り、話しかけてきた相手が本当に root かをどう確かめるか。
次に、ファイルを symlink やすり替えに騙されずに開く手順。
最後に、書いたものが途中で壊れにくく、しかも古い状態への一部の戻り方を防ぐための書き方である。

## 6.1 プロセス間通信と Unix domain socket

別々のプロセスが情報をやり取りすることを、**プロセス間通信**（IPC、inter-process communication）と呼ぶ。
bifrauth では、PAM モジュールと、root で動く判断役（verifier）が、別プロセスとして通信する。
このときに使うのが **Unix domain socket**（Unix ドメインソケット）である。

Unix ドメインソケットは、同じマシンの中のプロセス同士をつなぐ通信路である。
多くの場合、ファイルシステム上のパスを名前として持つ。
bifrauth では `/run/bifrauthd/pam.sock` のようなパスに作る。
接続を待ち受ける側（verifier）がそのパスにソケットを用意し、つなぎに来る側（PAM モジュール）が同じパスへ接続する。

このソケットも、第 1 章で見たファイルの一種として、所有者とパーミッションを持つ。
だから第一の壁として、ソケットを root 所有にし、一般ユーザーからは接続できない権限で作る。
〔仕様（正本）§11.1〕
bifrauth はこのソケットを root 専用とし、一般ユーザーから接続不可にする。
（このソケットを実際に誰が作り、所有し、どのモードで用意するか、という実装上の責務は第 19 章で扱う。bifrauth では systemd のソケットアクティベーションがソケットを用意する構成を取る。）

ただし、パーミッションだけを頼りにはしない。
設定ミスや、環境ごとの挙動の違いはありうる。
そこで、実際につないできた相手が誰なのかを、OS に直接尋ねる仕組みを重ねる。
それが次の `SO_PEERCRED` である。

## 6.2 SO_PEERCRED — つないできた相手を OS に問う

Unix ドメインソケットには、接続してきた相手のプロセスの素性を、OS へ問い合わせる仕組みがある。
それが **`SO_PEERCRED`** である（Linux 固有の機能）。
これを使うと、つないできた相手の UID、PID、GID を得られる。

肝心なのは、この値を埋めるのがカーネルだという点である。
相手のプロセスが自分で「私は root です」と名乗るのではない。
接続が成立したときの相手の資格情報を、カーネルが記録して返す。
だから、非特権のプロセスがこの値を偽って root のふりをすることはできない。

〔仕様（正本）§11.1〕
bifrauth の verifier は、PAM モジュールがつないでくると、まず `SO_PEERCRED` で相手の UID を確かめる。
それが root でなければ、信頼できる PAM の呼び出し元とはみなさない。
第 3 章で「bifrauth は信頼できる呼び出し元が渡す `PAM_SERVICE` を用途の根拠にする」と書いた、その「信頼できる呼び出し元」であることを、ここで確かめている。

いくつか、正確に押さえておくべき性質がある。
`SO_PEERCRED` が返すのは、接続が成立した時点の相手の資格情報である。
相手がその後で資格情報を変えても、この値は追随しない。
また、PID は使い回される。
あるプロセスが終われば、同じ PID が別のプロセスへ割り当てられうるので、PID を長期的な身元として当てにはできない。
認可の判断に使うのは、主に UID のほうである。

ここで、認証と認可の区別を丁寧に置いておく（第 2 章）。
`SO_PEERCRED` そのものは、相手の資格情報を提供するだけの仕組みである。
これは、相手が誰かを確かめる材料、つまり認証の側にあたる。
だが bifrauth が「UID が root（`0`）なら接続を受け付ける」と判断する部分は、その材料を使った**接続レベルの認可**である。
接続を許すかどうかを、資格情報に基づいて決めているからだ。
実際、この判断を行うコード（`serve.rs`。第 19 章）も、これを認可として扱っている。
つまり `SO_PEERCRED` は認証の材料を与え、それを使った接続レベルの認可がここで下される。
そのうえで、「root で接続してきた相手に、個々の要求や操作を許してよいか」という、より細かい操作単位の認可は、また別の層の判断になる（この先の各章）。

もう一点、この確認が証明する範囲にも線を引く。
`SO_PEERCRED` が示すのは「相手が root の資格情報を持つ」ことであって、「相手が `pam_bifrauth` という特定のバイナリだ」ことではない。
root で動くプロセスなら、どれであってもこの確認は通る。
これで十分なのは、bifrauth が root 側を信頼できる前提（TCB）に置いているからである（第 2 章）。
root がすでに侵害されている状況は、そもそも防御対象の外にある（正本設計書 §4.2）。
だから第 3 章で言った「信頼できる PAM の呼び出し元」とは、「root の資格情報を持つ相手」という意味であり、それ以上に特定のバイナリを暗号学的に見分けているわけではない。

## 6.3 ファイルを安全に開く — TOCTOU と symlink

通信路の次は、鍵や登録情報を置くファイルである。
第 1 章で、ファイル単体を `root:root 0640` にするだけでは足りないと見た。
ここでは、その「足りない分」を、実際に安全に開く手順として組み立てる。

まず、攻撃の形を二つ思い出す。
一つは、途中のディレクトリやファイルを **symlink**（別の場所を指す名前）へすり替える攻撃である。
狙ったパスを開いたつもりが、攻撃者の用意した別の場所を開かされる。
もう一つは **TOCTOU**（time-of-check to time-of-use）である。
「このファイルは安全だ」と確認した後、実際に使うまでの隙に、対象をすり替えられる。
確認と使用の間に時間の隙間があると、そこを突かれる。

対策の考え方は、二つある。
第一に、パスを一気に開かず、構成するディレクトリを一段ずつ、安全に辿る。
第二に、確認は「パスの名前」に対してではなく、「開いた後のファイルそのもの」に対して行う。

〔仕様（正本）§8.3。実装は `registry.rs`〕
bifrauth のレジストリは、この考え方を次のように具体化している。
信頼できる起点のディレクトリから始めて、パスの各成分を `openat` で一段ずつ開く。
このとき `O_NOFOLLOW`（symlink を辿らない）と `O_DIRECTORY`（ディレクトリでなければ失敗）を付ける。
一段開くたびに、開いたそのもの（ファイルディスクリプタ）に対して `fstat` で種別・所有者・mode を検証する。
途中のディレクトリが group や world から書けるようになっていれば、そこで止める。
最後のデバイスファイルも `O_NOFOLLOW` で開き、通常ファイルであること、所有者と mode、`nlink`（ハードリンク数）が 1 であること、サイズが上限内であることを確かめてから、はじめて読む。

`nlink` を確かめるのは、想定外のハードリンク（同じ実体を指す別の名前）が張られていないかを見るためである。
そして、確認をすべて「開いた fd」に対して行うのがかなめである。
名前をもう一度開き直せば、その隙にすり替えられるかもしれない。
一度開いた実体をそのまま確認し、そのまま使えば、確認と使用の間の隙がなくなる。

## 6.4 永続化の安全性 — 壊れにくく、限られた巻き戻しを防ぐ

読むだけでなく、書くときにも罠がある。
登録情報のようなファイルを、その場で上書きするとどうなるか。
書いている途中でマシンが落ちれば、半分だけ書かれた壊れたファイルが残る。
運が悪ければ、次に読んだとき登録情報が壊れている。

これを避けるのが、**アトミックな公開**（atomic publish）という書き方である。
考え方はこうである。
本番のファイルを直接いじらず、まず同じディレクトリに一時ファイルを作ってそこへ書く。
書き終えたら、それを本番の名前へ **rename**（改名）する。
rename は、同じファイルシステムの中でなら（ここでは同じディレクトリなので満たされる）、ひとつの不可分な操作として、名前を古い実体から新しい実体へ切り替える。
だから読む側からは、切り替えの前（古い内容）か後（新しい内容）のどちらかしか見えず、半分書かれた状態は見えない。

〔仕様（正本）§8.3。実装は `registry.rs`〕
bifrauth は、この手順を次のように行う。
同じディレクトリに、推測されにくい乱数の名前で一時ファイルを作る（`O_EXCL` で、既存の名前とぶつかったら失敗させる）。
そこへ書いてから `fsync` で確実にディスクへ書き出す。
それを本番の名前へ公開する。
登録のときは `renameat2` の `RENAME_NOREPLACE` を使い、同じ名前が既にあれば絶対に上書きしない（既にあれば「登録済み」として扱う）。
最後に、ディレクトリ自体も `fsync` する。

なぜファイルだけでなくディレクトリも `fsync` するのか。
rename は名前の張り替えなので、その事実はファイルの中身ではなくディレクトリに記録される。
ファイルの中身を `fsync` しても、そのディレクトリを `fsync` しなければ、改名がディスクに残らないまま落ちることがある。
一般化して言えば、新しい名前を作ったり rename したりした「そのディレクトリ」を、そのつど `fsync` する必要がある。

ここで、実装済みの範囲を正直に切り分けておく。
〔仕様（正本）§8.3。実装は `registry.rs`〕
bifrauth は、既にあるデバイスディレクトリの中でレコードを公開するときは、一時ファイルの `fsync` と、そのデバイスディレクトリの `fsync` を行う。
その範囲の耐久性（durability）は実装済みである。
一方で、初回の登録で `/etc/bifrauthd/users/<uid>/devices` という階層を新しく作る場合、その途中で作った親ディレクトリ群までを `fsync` する処理は、現状では入っていない。
だから「階層の新規作成まで含めた crash consistency」は、まだ実装済みとは言えない。
一般論（新しい名前を作った各ディレクトリを `fsync` する）と、現実装の到達点（既存 devices ディレクトリ内の公開は file＋dir の `fsync` 済み、新規階層の親 `fsync` は未実装）は、分けて理解しておく。
（なお、`fsync(dir)` が失敗したときは、名前の公開自体はすでに起きているため、単に「未適用」で片づく話ではなく、耐久性が不確かなまま公開された、という扱いになる。）

そして、もうひとつ大事な性質がある。
状態を古い方へ巻き戻す動きを、ある範囲で防ぐことである。
そこで bifrauth は二つの手を打つ。
一つ、失効を **tombstone**（墓標）として扱う。
失効はレコードを消すのではなく、「失効した」という印を付ける。
だから、正規の管理 API を通じて同じ端末 ID を後から登録し直して復活させる、ということはできない（第 18 章）。
一つ、壊れた・読めないレジストリを、黙って作り直さない。
読めないなら、勝手に空の状態から再生成せず、**fail closed**（安全側に倒して拒否する）で止まる。
黙って作り直せば、それは失効情報ごと消してしまう「巻き戻し」になるからだ。

ただし、この二つが防ぐ「巻き戻し」の範囲には、はっきり線がある。
防げるのは、正規の管理 API を通じた失効済み端末の復活と、破損を口実にした自動初期化である。
防げないのは、レジストリを書ける主体（root や、ディスクに触れるオフラインの攻撃者）が、以前に保存された**本物の古いスナップショット**をまるごと戻す巻き戻しである。
これを検出するには、単調に増えるカウンタや、外部に置いた基準（アンカー）、認証付きの世代番号といった仕組みが要るが、現状はそれを持たない。
そして root による書き換えは、そもそも防御対象の外である（正本設計書 §4.2）。
つまり tombstone と fail closed は、正規経路での復活と偶発的・不正入力による状態リセットを防ぐものであって、一般的な意味でのロールバック耐性まで保証するものではない。

## 6.5 三つの壁

この章で組み立てたのは、第 1 章の「root だけが書ける」を実際に効かせるための、三つの壁である。

- **誰がつないできたか**：ソケットを root 専用にし、さらに `SO_PEERCRED` の資格情報で「root なら接続を受ける」という接続レベルの認可を下す（root peer であることまでで、特定バイナリの証明ではない）。
- **安全に開く**：パスを一段ずつ `openat`／`O_NOFOLLOW` で辿り、開いた fd に対して種別・所有者・mode を検証する。
- **壊れにくく、限られた範囲で巻き戻しにくく書く**：一時ファイル → `fsync` → アトミックに rename → ディレクトリ `fsync`（既存ディレクトリ内の公開まで実装済み）、失効は tombstone、壊れたら fail closed（正規経路の復活と自動初期化は防ぐが、root/ストレージ層による本物の旧 snapshot 復元までは防がない）。

三つに共通するのは、「名乗り」や「名前」を信じず、実体を OS に確かめるという姿勢である。
相手の自己申告ではなくカーネルが記録した資格情報を、パスの名前ではなく開いた実体を、上書きの成否は運任せではなく rename の不可分性を頼る。

ここまでで、置き場所と読み書きの安全は固まった。
だが、そこへ置くバイト列そのもの、そして署名する対象そのものが、実装ごとにぶれてしまっては元も子もない。
第 5 章で「署名する側と検証する側はまったく同じバイト列を対象にしなければならない」と伏線を置いた。
次の章では、その「同じバイト列」をどう保証するか、正規化と CBOR の話へ進む。

## 理解度クイズ

> 解答と解説は付録 B にまとめてある。まず自分で考えてから確認してほしい。

1. Unix domain socket とは何をつなぐ通信路か。bifrauth は PAM モジュールと verifier の通信に、どのパスのソケットを使うか。
2. ソケットを root 所有・一般ユーザー接続不可にしても、bifrauth がそれだけに頼らず `SO_PEERCRED` を重ねるのはなぜか。
3. `SO_PEERCRED` が返す相手の UID/PID/GID は、誰が埋める値か。非特権プロセスがこれを偽って root のふりをできないのはなぜか。
4. `SO_PEERCRED` について、(a) 返す資格情報はいつの時点のものか、(b) PID を長期的な身元として当てにできない理由、を述べよ。
5. `SO_PEERCRED` は認証と認可のどちらの材料か。bifrauth が「UID が root なら接続を受け付ける」と判断する部分は、認証・認可のどちらにあたるか。また、この確認は「相手が `pam_bifrauth` という特定のバイナリだ」ことまで証明するか。
6. ファイルを安全に開くとき、「パスの名前」に対して確認するのと「開いた後のファイル（fd）」に対して確認するのとでは、なぜ後者でなければならないか。TOCTOU に触れて説明せよ。
7. bifrauth のレジストリがパスを辿るとき `openat` に付ける `O_NOFOLLOW` と `O_DIRECTORY` は、それぞれ何を防ぐ・保証するためか。最後に読むファイルで `nlink==1` を確かめるのは何のためか。
8. 登録情報を「その場で上書き」せず、一時ファイル → rename で公開するのはなぜか。rename の性質に触れて説明せよ。
9. 新しい名前を作ったり rename したりしたとき、その中身を `fsync` するだけでなく、その名前を持つディレクトリも `fsync` する必要があるのはなぜか。bifrauth の現実装で、この `fsync` が「既存の devices ディレクトリ内の公開」については済んでいるが「初回の階層の新規作成」については未実装、という切り分けも述べよ。
10. 失効を tombstone で行い、壊れたレジストリを fail closed にすることで防げる「巻き戻し」は何か。逆に、これらでは防げない「巻き戻し」は何か（root やオフライン攻撃者を念頭に）。
