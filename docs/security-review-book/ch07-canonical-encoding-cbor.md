# 第 7 章 バイト列を一致させる技術 — 正規化・CBOR・バージョン付け

> **この章の実装状態:** 正規化と CBOR の一般論を扱う。bifrauth 固有の記述には節ごとにラベルを付ける。
> BifrAuth CBOR プロファイルとその検査は〔仕様（正本）§9.4、`spec/cbor-profile.md`。実装は
> `bifrauth-proto`（`cbor.rs` / `schema.rs` / `text.rs`）で〔実装済み〕〕。版まわりで実装済みなのは、
> 各メッセージ族につき v1 のみ（`protocol_version = 1`）の完全一致検査・応答と保留 challenge の版一致・
> 署名対象への版の束縛であって、**複数版を広告して選ぶ「交渉」は未実装（将来課題）**（§7.6）。
> 参照時点: commit `826937b`（2026-07-20 JST）。

第 5 章で、伏線を一つ置いた。
署名する側と検証する側は、まったく同じバイト列を対象にしなければならない、という約束である。
一文字でも、並び順が一つでも違えば、署名は通らない。
だが、そこで一つ疑問が残る。
別々に作られた二つの実装（Linux 側の Rust と、iPhone 側の Swift）が、同じ意味のデータから本当に一致するバイト列を作れるのか。

作れる、と言い切るには、仕掛けが要る。
この章では、まず「同じ意味なのにバイト列が違う」がなぜ起きるかを見る。
次に、その違いを消すための決まり（正規化と決定論的 CBOR）と、そこに潜む落とし穴をたどる。
そして、未知の項目や重複をどう扱うか、受け取ったバイト列をどう検査するか、さらに版をどう付けるか（いまの厳格な版検査と、将来の downgrade 対策という課題）へ進む。

## 7.1 なぜ「同じ意味」でもバイト列が食い違うのか

同じ内容のデータでも、書き表し方は一通りではない。
たとえば、二つの項目を持つ表を考える。
一方の実装は「名前、期限」の順に並べ、もう一方は「期限、名前」の順に並べるかもしれない。
数値の `1000` を、ある実装は詰めた形で、別の実装は余分な桁を付けた形で書くかもしれない。
文字列が「同じ文字」でも、内部のバイトの並びが違うことすらある（後で見る Unicode の正規化の話。ここでいう「同じ文字」とは、見た目が似ているという意味ではなく、Unicode 上で同一と定められた文字の別表現のことである）。

人間の目にはどれも「同じ意味」に見える。
だが、署名はバイト列に対して計算される。
意味が同じでもバイト列が違えば、ハッシュが変わり、署名は一致しない。
つまり、意味の一致では足りず、**バイト列の一致**（byte-exact match）が要る。

さらに厄介なのは、この食い違いが安全性の穴にもなることである。
同じ入力を二つの実装が別々に解釈できるなら、攻撃者はその差を突ける。
一方が無視する項目を、もう一方が意味あるものとして受け取る、といったずれが起きれば、そこが抜け道になる。
だから、表現を一つに定める必要がある。
この「同じ意味は必ず同じバイト列になる」ようにエンコーディングの表現を一意に固める作業を、**正規化**（canonicalization）と呼ぶ。
（紛らわしいが、これは後で §7.3 に出てくる Unicode の正規化（Unicode normalization）とは別の概念である。前者はバイト列の書き方の一意化、後者は文字列の表現の一意化を指す。本書では区別して扱う。）

## 7.2 CBOR と決定論的エンコーディング

bifrauth が正規化の土台に選んだのは、**CBOR**（Concise Binary Object Representation、簡潔なバイナリオブジェクト表現）である。
CBOR は、JSON に似た構造（数値・文字列・配列・map など）を、テキストではなくバイナリで表す形式である。
JSON と違い型が明確で、コンパクトに詰められる。

ただし、CBOR というだけでは足りない。
素の CBOR は、同じ値を複数のバイト列で書けてしまう。
そこで、書き方を一通りに絞った **決定論的 CBOR**（deterministic CBOR）を使う。
その基礎は RFC 8949 §4.2 が定める core deterministic encoding で、要点は次のようなものである。

- **長さは必ず確定長で書く。** 「これから何個」を先に示す形にし、終端記号で締める可変長（indefinite length）は使わない。
- **最短の形で書く（preferred serialization）。** 同じ整数や長さを表す複数の書き方のうち、いちばん詰めた一通りだけを許す。
- **map のキーは決まった順に並べる。** どの実装も同じ順序になるよう、キーの順序を規則で固定する。

これらを守れば、同じ値からは同じバイト列が出る。
Linux 側と iPhone 側が別々に実装しても、バイト列が一致する見込みが立つ。

## 7.3 正規化の落とし穴と、BifrAuth プロファイル

RFC の core deterministic encoding は土台だが、それだけでは実務で足りない。
現実の落とし穴を、さらにふさぐ必要がある。
〔仕様（正本）§9.4、`spec/cbor-profile.md`〕
bifrauth は、この上に **BifrAuth CBOR プロファイル**という追加の制約を重ねている。
主な落とし穴と、その対策を並べる。

- **キーの順序**：map のキーを厳密に昇順（前のキー < 次のキー）に固定する。順序が曖昧だと、同じ内容が別のバイト列になる。bifrauth は map のキーを符号なし整数だけに限っているので、エンコードしたバイト列を辞書順に並べる規則が、そのままキーの数値の昇順と一致する。
- **重複キー**：同じキーが二度現れるのを拒否する。放置すると、どちらを採るかが実装ごとに割れ、解釈の食い違いを生む。
- **数値の表現**：最短形以外を拒否する。`1000` を余分な桁で書いた非最短のバイト列は、受け付けない。
- **使ってよい型を絞る（allowlist）**：使うのは符号なし整数・バイト列・文字列・map・`null` だけとする。浮動小数点数（float）、タグ（tag）、真偽値（bool）などは、そもそも登場させない。これらが正規化できないからではない（RFC 8949 には float の決定的規則もある）。bifrauth に不要な表現や意味論を持ち込まず、実装差が出る面積を減らすために、必要な型だけへ絞る判断である。
- **文字列の正規化（NFC）**：文字列は UTF-8 かつ **NFC**（Unicode 正規化形式 C、Normalization Form C）に統一する。これは、Unicode 上で同一と定められた文字の別表現（たとえば合成済みの「é」と、「e」に結合用のアクセント記号を続けた分解表現）を、一つのバイト列へ揃えるものである。見た目が似ているだけの別の文字（ラテン文字の a とキリル文字の а など）まで統一するわけではない点に注意する。これは application 側の制約であり、RFC core の要件ではない（詳しくは §7.5）。
- **サイズの上限**：各フィールドとメッセージ全体に、あらかじめバイト数の上限を設ける。過大な入力でメモリを食いつぶす攻撃（後の章の resource limit の話）を、確保の前に弾く。

このプロファイルは、メッセージの意味に依存しない層（バイト列がプロファイルの決定的規則に従うか）と、メッセージ種別ごとのスキーマ（どのキーにどの型が入るか）の、二つの層に分けて定義されている。
前者を通ってから、後者で「この種類のメッセージとして正しいか」を見る。

## 7.4 未知フィールドと重複は fail-closed で拒否する

ここで、設計の姿勢がはっきり出る点がある。
知らないキーや重複キーに出会ったとき、どうするか。

素朴には、「知らないキーは無視して、分かるものだけ使えばよい」と考えたくなる。
多くの一般的なデータ処理では、それが寛容で良い作法とされる。
だが認証では、それは穴になる。

なぜなら、無視するという振る舞いが、実装ごとにずれるからだ。
一方の実装が黙って無視するキーを、もう一方が意味あるものとして受け取れば、二つは「同じメッセージ」を違うものとして扱う。
攻撃者は、その差へ細工を忍ばせられる。
重複キーも同じで、どちらの値を採るかが割れれば、そこが抜け道になる。

だから bifrauth は、逆の姿勢を取る。
知らないキー、余分なキー、重複したキーに出会ったら、無視せず **拒否する**。
安全側に倒して止まる、この振る舞いを **fail-closed** と呼ぶ。
必須のキーが欠けていても、同じく拒否する。
「分かるものだけ拾う」寛容さではなく、「一字一句、決めたとおりでなければ通さない」厳格さを選ぶ。
曖昧さを残さないことが、実装間の解釈のずれを消し、そのずれを突く攻撃をふさぐ。

## 7.5 受け取ったバイト列そのものを検査する

正規化を実装するとき、素朴なやり方が一つある。
受け取ったバイト列をいったんデコード（解釈）し、それを自分で再エンコード（書き直し）して、元のバイト列と比べる、という方法である。
一致すれば「正規だった」とみなす。

この「デコードして再エンコードして比べる」方式も、厳密にやれば正規性の判定として成立はしうる。
元のバイト列と、正しい正規エンコーダの出力とを厳密に比べれば、非正規な入力はふつう弾ける。
問題は、方式が必ず誤りだということではなく、依存の広さである。
寛容な（あるいは情報を落とす）デコーダは、重複のような情報をこっそり捨ててしまう。
比較の基準がエンコーダやライブラリの挙動にまで及ぶので、「何を受理し何を拒否するか」を証明しにくくなり、レビューすべき対象と信頼の土台（TCB）が広がる。
入力のサイズ上限を、デコードの前に効かせるのも難しい。

そこで bifrauth は、別の発想を取る。
〔仕様（正本）§9.4、実装は `bifrauth-proto`〕
受け取ったバイト列を書き直して比べるのではなく、**受け取ったバイト列そのものが正規かどうかを直接検査する**。
しかも、これは二段構えになっている。
一段めは、構造の直接走査である（`cbor.rs`）。
バイトを先頭からたどりながら、確定長になっているか、最短形か、キーが昇順か、重複がないか、許された型か、正しい UTF-8 か、を確かめる。
二段めは、その値についての意味の検査である（`schema.rs` から `text.rs`）。
必須のキーがそろっているか、余分なキーがないかを確かめ、文字列については text policy として、NFC であること・未割当の文字を含まないこと・制御文字や双方向制御文字を含まないことを、それぞれ検査する。
NFC の判定は、この二段めの責務であって、一段めの構造走査には含まれない。
どちらの段でも、規則に反していればその時点で拒否する。

この発想は、レビューでも効く。
「デコードできたか」ではなく「受け取ったバイト列は規則どおりだったか」を問う。
寛容なデコーダは、非正規な入力を通してしまいがちだが、直接検査ならそれを取りこぼさない。

## 7.6 バージョン付けと downgrade — いま守れること、将来の課題

最後に、時間軸の話を加える。
プロトコルは、いつか版が上がる。
新しいメッセージ形式や、より強い署名方式が加わるかもしれない。
このとき気をつけるべき攻撃が、**downgrade 攻撃**（downgrade attack）である。
攻撃者が、通信を古い版や弱い方式へ引き下げさせ、そこにある弱点を突く、という手口である。
第 5 章で署名の話をしたが、この版・方式の選択は、TLS のような通信路だけの問題ではなく、どの形式・方式・スキーマで話すかというプロトコル全体の問題である。

〔仕様（正本）§9.4、`spec/cbor-profile.md` §10〕
bifrauth は、版の扱いに役割の分離を設けている。
スキーマ（メッセージの中身の構造）を変えるときは、必ず新しいメッセージ種別（`message_type`。たとえば `bifrauth.challenge.v2`）にする。
同じメッセージ種別のスキーマを、版番号の値で切り替えることはしない。
一方で、全体の互換セットを表す番号（`protocol_version`）は、別の役割として持つ。
版の束縛のされ方には、二つの経路がある。
challenge 側の版（`protocol_version`）は canonical challenge の中に入っているので、Ed25519 と P-256 の署名対象にそのまま含まれ、直接束縛される。
応答側の版は、署名された保留 challenge の版と一致するかを検査することで、間接的に束縛される。
古いスキーマのキーは、二度と再利用しない。

ここで、実装されていることと、まだ設計されていないことを、はっきり分けておく。
いま実装されているのは、次のことである。
対応する唯一の版（v1）に完全一致するかの検査。
応答の版が保留中の要求の版と一致するかの確認。
そして、版が署名対象に束縛されていること。
対応版が一つしかない現在は、未知の版や改変された版を拒否できる。

だが、これは複数版を広告して選ぶ「交渉」ではない。
将来 v1 と v2 を併用するようになったとき、攻撃者が「v2 に対応している」という事実を隠して v1 へ落とさせる攻撃を、いまの仕組みは防げない。
「downgrade を許さない」という要求は方針であって、それ自体は機構ではない。
複数版を安全に扱うには、別途の設計が要る。
たとえば、双方が対応する版の集合と、選んだ結果とを、署名などで認証の対象へ束縛すること。
そして、受け入れてよい最低の版を定めるポリシーを持つこと。
これらは将来の課題である。

もう一つ、例外を押さえておく。
challenge や response は、いま述べたとおり新しいメッセージ種別で版を分ける。
だが challenge を運ぶ封筒（envelope）は、通信の上に `message_type` や `protocol_version` を持たない。
その版は、外側の枠組み（framing）で固定される（`spec/cbor-profile.md` §6・§10）。
本章の一般則（スキーマを変えるなら新メッセージ種別）と、この envelope の扱いは、区別しておく。

この「版と方式を弱いほうへ動かさせない」姿勢は、いまは限られた形でしか実現されていないが、狙いは署名対象への文脈の束縛（第 5 章）と同じである。
何をもって正しい要求とするかを、あらかじめ一意に固め、そこからのずれを拒否する。

## 7.7 ここまでの位置づけ

第 5 章から続く二章で、署名が効くための土台が固まった。
第 5 章は「同じバイト列に署名・検証する」と約束し、この章はその「同じバイト列」を、実装が違っても作れるように定めた。
正規化で表現を一意にし、未知・重複・非正規を fail-closed で拒否し、受け取ったバイト列そのものを検査した。
版については、現行 v1 では未知・改変された版を拒否できるが、複数版を併用するときの downgrade 防止は将来の課題として残る（§7.6）。

ここまでは、一つの要求が「正しい形か」を確かめる話だった。
次の章では、時間の軸を加える。
その要求が「今このためのものか」を確かめる話、すなわち時刻と有効期限、状態機械、そして一度きりの消費へ進む。

## 理解度クイズ

> 解答と解説は付録 B にまとめてある。まず自分で考えてから確認してほしい。

1. 「意味が同じでもバイト列が違う」ことが、なぜ署名にとって問題になるのか。また、それが単なる不便ではなく安全性の穴にもなりうるのはなぜか。
2. 素の CBOR ではなく決定論的 CBOR を使うのはなぜか。決定論的 CBOR の要点を三つ挙げよ（確定長・最短形・キー順序など）。
3. BifrAuth プロファイルが「使ってよい型を絞る（allowlist）」で float や tag を閉め出すのは、何を防ぐためか。
4. 認証において「知らないキーは無視して、分かるものだけ使う」寛容な作法が穴になるのはなぜか。bifrauth はそれをどう扱うか（fail-closed）。
5. 重複キーを拒否するのはなぜか。放置すると何が起こりうるか。
6. 「デコードして再エンコードし、元のバイト列と比べる」方式は、必ず誤りだと言えるか。この方式の問題（依存の広さ・TCB の拡大）を述べ、bifrauth が代わりに選ぶ方式を、二段構え（構造の直接走査と、値についての意味の検査）に触れて説明せよ。また NFC の検査は、この二段のどちらの責務か。
7. downgrade 攻撃とは何か。bifrauth で「いま実装されていること」（v1 完全一致検査・版の一致確認・署名への版の束縛）と、「まだ設計されていないこと」（複数版を広告して選ぶ交渉）を分けて述べよ。将来 v1/v2 を併用するとき、いまの仕組みが防げない攻撃は何か。
8. bifrauth は、challenge や response のスキーマを変えるときにどうするか。また、challenge を運ぶ封筒（envelope）の版は何によって固定されるか（一般則の例外として）。
9. 文字列を UTF-8 かつ NFC に統一すると、どんな食い違いを防げるか。逆に、NFC では防げない「見た目の類似」（別の文字なのに似て見える場合）があることにも触れよ。
