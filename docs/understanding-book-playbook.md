# 理解を助ける本の作り方（プロジェクト非依存プレイブック）

> **これは何か:** ユーザーが知識の乏しい分野のコードを扱うとき、「その分野を学べる本」を
> 書きながらコードの理解とセキュリティレビューを可能にする、という進め方の**再利用可能な
> ノウハウ集**。特定プロジェクトに依存しない形で書く。
>
> **育て方:** bifrauth 学習本（`docs/security-review-book/`）を実際に作りながら、うまくいった
> 型・つまずき・判断基準を随時ここへ追記する。将来この内容が安定したら Claude Code スキル化する
> （まずリポジトリ内ノートとして育て、抽出はあとで）。
>
> **ステータス:** 作成中（bifrauth 第1章の執筆時点で起筆）。

---

## 0. この進め方が有効な状況

- ユーザーがドメイン知識の乏しい分野のコードを、エージェントに書かせた／書かせたい。
- ユーザー自身がそのコードを理解し、できれば自分でレビュー（特にセキュリティ）できるようになりたい。
- 「コードだけ渡す」「口頭で説明する」では、前提知識の欠落が埋まらない。

このとき、**その分野の入門から実コード解説までを一冊の本として書き、章ごとに検証しながら進める**と、
ユーザーの理解が段階的に積み上がり、レビューの土台ができる。

---

## 1. 全体の型

### 1.1 二部構成（＋実践編）

本は大きく次で構成する。

1. **プロジェクト非依存の技術解説章（複数）** — その分野の前提を、初学者向けに順序立てて。
2. **実プロジェクトのコード解説章（複数）** — 実装をコンポーネント単位で読み解く。
3. **（レビュー目的なら）実践編** — 脅威モデル作成 → 不変条件で追跡 → 敵対的テスト → finding 報告。

### 1.2 進め方のパイプライン

1. **調査**: 設計書・実装・既存レビュー記録を読み、地図を持つ（章構造の精度がここで決まる）。
2. **章構造案（アウトライン）を先に出す** → クロスレビュー＋ユーザー承認。
3. 技術解説章を順に執筆（**章ごとにクロスレビュー**）。
4. コード解説章・実践編を順に執筆（**章ごとにクロスレビュー**）。

各段階で「確定してから次へ」。アウトライン未承認のまま本文を書かない。

## 2. 章構造（アウトライン）の設計原則

- **前提ゼロから積み上げる順序にする。** 応用（例: PAM/polkit）より先に、その土台（OS の最小モデル、
  ユーザー/プロセス/権限/ファイル）を置く。「後で出てくる言葉」を先に説明しておく。
- **章数を先に固定しない。** 必要なトピックを洗い出してから章に割る。「10章に収める」より
  「抜けを作らない」を優先する。
- **運用・非機能の章を忘れない**（レビュー目的なら特に）: 可用性/レート制限/DoS、ログと監査/プライバシー、
  バージョン交渉/downgrade、永続化の安全性、FFI 境界、テスト技法。機能フローだけだとレビューに不足する。
- **コード解説章の割り方は実装の責務境界に合わせる。** ファイル名で章を切らず、「このファイルが実際に
  担う責務」で切る（1ファイルに複数責務があれば分割、逆に統合できるものはまとめる）。
  この精度はコードを実際に読んでからでないと出せない。
- **実践編は薄くしすぎない。** チェックリスト一枚で終えず、脅威モデル作成→不変条件追跡→敵対的テスト→
  finding 報告、と手を動かす章に分ける。チェックリストは最後の成果物にする。

---

## 3. 正確さを守る仕掛け（レビュー用途では必須）

### 3.1 「正しさの順位」（source of truth）を冒頭で固定する

矛盾を見つけたときにどれを正とするかの順位を、本の最初に明記する。典型は次。

1. 正本設計書（唯一の正）
2. 正本から明示的に委譲・参照された詳細仕様（委譲範囲内でのみ規範的、正本を上書きしない）
3. 実装（＝仕様適合を検査する対象）
4. 本書（読み解くための導き）

矛盾時は本書を根拠に実装や正本を書き換えず、**finding として報告**する。古い設計文書は参照禁止にする。

### 3.2 実装状態ラベルを各章・各節に付ける

`仕様（正本）` / `実装済み` / `mockのみ` / `未実装・将来予定` を、参照 commit/date とともに明記する。
**「未実装を安全性の証明に数えない」「mock を読んでも本番をレビューしたことにならない」**を繰り返し教える。
時点依存の状態は、可変な進捗ファイルの文言に頼らず、執筆対象 commit/date のラベルで固定する。

### 3.3 コード解説章の定型枠

各コード章に固定枠を置く: **正本の対応節 / 主要 invariant / source files（commit permalink）/ tests /
未実装・前提**。読者が毎回同じ視点でコードを読めるようにする。

### 3.4 確定していないことを確定として教えない

設計上まだ未決の選択（例: 通信方式の候補が複数ある）は、「目的」「候補」「未決である事実」を分けて書く。
詳細計画が暫定的に一方へ寄っていても、正本が未確定なら「暫定選好」と明示する。
採用済み仕様・候補・一般論を混同しない。

### 3.5 「危険な単純化」を避ける（クロスレビューが効く箇所）

初学者向けの簡略化は必要だが、**セキュリティの結論を誤らせる単純化はしない**。実例（bifrauth 第1章で
クロスレビューが捕捉したもの）:

- 「root だけが書けるファイル」だけでは改ざん防止にならない（親ディレクトリの保護、symlink/TOCTOU、
  安全な open まで含めて初めて壁になる）。
- 「プロセス＝起動ユーザーの UID」は、effective UID/GID・capability・setuid/sudo 等の境界を添える。
- OS の「root は何でもできる」は、capability/LSM/namespace/immutable 等の留保を短く付ける。

簡略化するときは「本書ではこう略す」という但し書きを置き、略した先に何があるかを一言示す。

## 4. 各章の書き方

- **章末に理解度クイズ**を置く。問題は章末、**解答・解説は付録に集約**（二重管理を避ける）。解答には
  根拠（本文の節・設計書の該当箇所）へのポインタを付ける。
- クイズは「本文の重要点を再確認する」問いにする。特に安全性の勘所（誰が信頼され、何が壁か）を問う。
- 用語は**初出で定義（太字）**、以降は通称。序章は索引・最小語、正式な用語集は付録に。
- 文章規範（日本語なら一文一行・パラグラフライティング・LLM っぽい空句の排除など）に従う。

### 4.1 読み進められる文章にする（認知リズム）

正確なだけでは読み進められない。技術解説でも、読者の認知モード（観察→逡巡→確信→再確認）を
切り替えて推進力を作る。説明文で使える具体策:

- **素朴な期待を一度肯定してから、事実で崩す。** 初学者が当然抱く結論（例:「ファイルを `root:root
  0640` にすれば改ざんされない」）をいったん確信の声で書き、直後に反例（親ディレクトリ・symlink・
  TOCTOU）で崩す。この「崩れ」が次を読む動機になる。§3.5 の「危険な単純化」の是正が、そのまま良い
  リズムの素材になることが多い。
- **文の拍を変える。** 短文で足場を打ち、長文で流し、短文で止める。断定ばかり続けない。
- **段落の密度に波をつける。** 密な段落が続いたら、確定事項を一行で固定する疎な段落を挟む。
- **箇条書きは場面へ着地させる。** 性質を列挙したら「これは冒頭の失敗のどれに当たるか」を一つずつ結ぶ。
- **節の橋は次節の頭に置く。** 節末に「次は〜を見る」と予告しない。次節を反問や違和感で開始する。
- **やってはいけない: 本文自身を話題にした文でリズムを作る**（「ここでは列挙しない」「本章の主題は
  〜である」等）。緩急の材料は対象の性質と読者の推論からのみ取る。書いた文が「状況を更新するか、
  文書を更新するか」で判定し、文書更新だけの文は削る。

## 5. クロスレビュー運用（もう一体のエージェントと）

- **各成果物（アウトライン、各章）を、もう一方のエージェントにレビューさせる。** 一人で書いて一人で
  確定しない。特に §3.5 の「危険な単純化」は自分では気づきにくく、レビューがよく捕捉する。
- **最大3ラウンド**で合意を目指す（指摘反映→再レビュー を1ラウンド）。3ラウンドで合意できなければ
  ユーザーへエスカレーション（争点・双方の主張・往復の要点・選択肢を添える）。
- レビュー観点は「事実誤り／危険な単純化」「順序の飛躍」「実装との不一致」「正本との整合」「クイズの妥当性」。
- レビュー依頼・返信の往復記録が、そのまま品質の証跡になる。

---

## 6. 運用まわり（実務メモ）

- 本作りは長期タスクになるので、**タスク管理に登録**して進捗ログを残す。
- 相手エージェントが在席でも idle なら、依頼を送っただけでは読まれないことがある。起こす手段
  （このリポジトリでは `herdr pane run`）を用意しておく。自分側は受信を監視していれば返信は自動で届く。
- 章・付録はファイルを分け、付録（クイズ解答・用語集）は章と並行して更新する（書き忘れ防止）。
